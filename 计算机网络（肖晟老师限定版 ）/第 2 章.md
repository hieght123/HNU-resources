# 2.1 应用层协议原理
## 2.1.1 网络应用体系结构
### 重要性
★★★
### 要背的边角料
- **客户-服务器 (C/S) 体系结构**: 
    - 服务器：总是打开，具有固定 IP (网际协议) 地址，通常在数据中心。
    - 客户：间断连接，IP 地址可变，不直接与其他客户通信。
- **对等 (P2P) 体系结构**: 
    - 间断连接的主机对直接通信。
    - **自扩展性 (Self-scalability)**：每个对等方在请求文件的同时也向其他对等方分发，增加系统能力。
- **网络体系结构 vs 应用程序体系结构**: 网络体系结构指协议栈分层；应用程序体系结构指应用进程在端系统中的组织方式，如 C/S 或 P2P。
- **非专用因特网应用与协议例子**: Web (万维网) 使用 HTTP/HTTPS (超文本传输协议/安全超文本传输协议)；电子邮件使用 SMTP (简单邮件传输协议)、IMAP/POP3 (因特网邮件访问协议/邮局协议第 3 版)；文件传输使用 FTP (文件传输协议)；远程登录使用 SSH/Telnet (安全外壳协议/远程登录协议)；域名解析使用 DNS (域名系统)。
- **即时通信与 SMS 边界**: iMessage、WhatsApp、微信等即时通信应用通常运行在因特网应用层协议之上，经 TCP/UDP/TLS 等承载；SMS 是蜂窝网络的短消息服务，不等同于这些互联网即时通信协议。

### 最可能怎么考
- **简答题**: 网络体系结构与应用程序体系结构有什么区别？
  - **答案**：网络体系结构讨论通信系统的分层协议组织，例如应用层、运输层、网络层等；应用程序体系结构讨论应用程序在端系统之间如何组织通信，例如客户-服务器体系结构和 P2P 体系结构。
- **简答题**: P2P 体系结构的主要优点是什么？
  - **答案**：
    1. **自扩展性 (Self-scalability)**：每个对等方在请求文件的同时也向其他对等方分发，服务能力随用户增加而增长。
    2. **成本效益**：降低了对中心服务器带宽的极高需求。

## 2.1.2 进程通信与套接字
### 重要性
★★★★
### 要背的边角料
- **套接字 (Socket)**: **应用程序与运输层之间的接口**。也被称为网络与应用程序之间的 **API (应用程序编程接口)**。
- **标识进程**: 需要 **IP 地址** + **端口号**。
- **客户进程/服务器进程判定**: 发起通信的是客户进程，等待被联系的是服务器进程；在 P2P 中，同一进程在不同通信会话中可以既充当客户又充当服务器。
- **🚨 408 必背熟知端口大全**:
    - **FTP**: 数据连接 20，控制连接 21 (TCP，传输控制协议)
    - **SSH**: 22 (TCP)
    - **Telnet**: 23 (TCP)
    - **SMTP**: 25 (TCP)
    - **DNS**: 53 (UDP为主，也有TCP)
    - **DHCP (动态主机配置协议)**: 客户端 68，服务器 67 (UDP，用户数据报协议)
    - **HTTP**: 80 (TCP)
    - **POP3**: 110 (TCP)
    - **IMAP**: 143 (TCP)
    - **HTTPS**: 443 (TCP)

### 最可能怎么考
- **填空题**: 进程通过一个称为 **套接字 (Socket)** 的软件接口在网络上发送和接收报文。
- **简答题**: 为什么在进行网络通信时，仅有目标主机的 IP 地址是不够的？
  - **答案**：因为一台主机上通常同时运行着多个网络应用程序（如 Web 浏览器和邮件客户端）。IP 地址只能将数据报路由到**特定的目的主机**，而**端口号**才能将数据准确地交付给该主机上对应的**特定应用程序进程**。
- **简答题**: 对于两个进程之间的通信会话，如何判断哪个是客户、哪个是服务器？
  - **答案**：在一次通信会话中，**发起通信**的进程是客户进程，**等待联系**的进程是服务器进程。在 P2P 应用中，一个对等方进程既可能向别人请求文件块，也可能向别人上传文件块，因此它在不同会话中可以同时体现客户和服务器角色。

## 2.1.3 可供应用程序使用的运输服务
### 重要性
★★★
### 要背的边角料
- **可靠数据传输**: 确保数据正确、完全交付。
- **吞吐量**: 确保可用吞吐量不低于某速率（带宽敏感 vs 弹性应用）。
- **定时**: 保证在特定时间内交付数据。
- **安全性**: 加密、数据完整性、端点鉴别。
- **既要求可靠又高度时间敏感的应用**: 远程手术、工业实时控制、自动驾驶远程控制等。这类应用既不能丢关键数据，也不能有明显时延。

### 最可能怎么考
- **简答题**: 区分带宽敏感应用与弹性应用。
  - **答案**：**带宽敏感应用**具有明确的吞吐量下限，若网络不能提供足够吞吐量，应用质量会明显下降甚至无法工作，例如因特网电话、视频会议、流式视频。**弹性应用**没有固定吞吐量下限，能够根据当时可用带宽或快或慢地传输，吞吐量越高体验越好，但吞吐量较低时仍可运行，例如电子邮件、文件传输和 Web 传送。

## 2.1.4 因特网提供的运输服务
### 重要性
★★★★
### 要背的边角料
- **TCP 服务**: 面向连接、可靠传输、拥塞控制、全双工。
- **UDP 服务**: 无连接、不可靠、无拥塞控制、轻量。
- **不提供的服务**: 当前因特网运输层不提供 **吞吐量** 或 **定时** 保证。
- **应用选择运输协议**: HTTP、SMTP、IMAP 需要可靠交付，通常运行在 TCP 之上；UDP 适合低开销、能容忍少量丢失或由应用自行处理可靠性的场景。
- **最快事务可选 UDP**: 如果远程事务只追求低时延、能容忍丢失或由应用自己处理可靠性，可选 UDP，因为它没有 TCP 三次握手；若要求完整、按序、可靠，则仍选 TCP。
- **TLS/DTLS/QUIC (传输层安全/数据报传输层安全/快速 UDP 因特网连接) 位置**: 传统 TLS 通常运行在应用层和 TCP 之间，依赖 TCP 的可靠字节流；若要强化 UDP，通常使用 DTLS，或使用基于 UDP 并内置安全机制的 QUIC。

### 最可能怎么考
- **填空题**: 当前因特网运输层协议 **不提供** 吞吐量或定时的保证。
- **简答题**: 为什么 HTTP、SMTP、IMAP 都运行在 TCP 而不是 UDP 上？
  - **答案**：这些应用传输的是 Web 对象或邮件内容，要求数据完整、按序、可靠到达。TCP 提供可靠数据传输、流量控制和拥塞控制，应用层不必自己处理丢失和乱序；UDP 不提供可靠交付，因此不适合作为这些协议的默认运输服务。

# 2.2 Web 和 HTTP
## 2.2.1 HTTP 概述
### 重要性
★★★★★
### 要背的边角料
- **HTTP**: 超文本传输协议，**无状态协议 (Stateless)**。
- **运输协议**: 使用 **TCP**。
- **Web 页面/对象**: Web 页面也叫文档，由多个对象组成；对象可以是 HTML (超文本标记语言) 文件、JPEG (联合图像专家组图像格式) 图形、JavaScript (脚本语言) 文件、CSS (层叠样式表) 样式表或视频片段。
- **基本 HTML 文件**: 多数 Web 页面含有一个 **base HTML file (基本 HTML 文件)** 和若干被它通过 URL 引用的对象；若 1 个基本 HTML 文件引用 5 个 JPEG 图形，则该页面共有 **6 个对象**。
- **URL (统一资源定位符) 两部分**: 存放对象的服务器主机名 + 对象路径名。浏览器是 HTTP 客户端，Web 服务器存储由 URL 寻址的 Web 对象。
- **TLS 位置**: 运行在 TCP 之上，但在应用层之下。
- **握手协议作用**: 在传输应用数据前交换控制信息，确认双方准备好通信并建立必要状态，例如 TCP 建连前的三次握手。

### 最可能怎么考
- **判断题**: HTTP 服务器会记录客户在几秒内重复请求同一个对象的状态吗？（不会）。
- **填空题**: 一个 Web 页面的基本 HTML 文件引用了 4 个图片对象，则该 Web 页面共有 **5 个对象**。

## 2.2.2 持续与非持续连接
### 重要性
★★★★★
### 要背的边角料
- **往返时间 (RTT)**: 一个短分组从客户到服务器再返回客户所花费的时间。
- **HTTP/1.0 默认**: **非持续连接 (Non-persistent)**。每个请求/响应对经一个单独的 TCP 连接发送。
    - 响应时间 = **2 RTT + 文件传输时间**（1个RTT建TCP，1个RTT发HTTP请求并接收响应前几个字节）。
- **HTTP/1.1 默认**: **持续连接 (Persistent)**。所有的请求及其响应经相同的 TCP 连接发送。
    - **流水线 (Pipelining)**: 不必等待未决请求的回答即可连续发出请求。
- **🚨 协议格式陷阱**: HTTP 请求报文中的每一行以 **CRLF (回车换行)** 结束；首部行与实体体之间用一个**只包含 CRLF 的空白行**分隔。

### 最可能怎么考
- **计算题/简答题**: 计算并对比获取一个带有 N 个小图像引用的 HTML 基础页面时，非持续连接和带流水线的持续连接分别需要多少时间？
  - **答案 (解析步骤)**：
    1. **非持续连接**：获取基础 HTML 需 2 RTT；后续每个图像均需新建 TCP，需 2 RTT。总时间 = 2 RTT + N * (2 RTT) = **(2N+2) RTT** + 传输时间。
    2. **带流水线的持续连接**：获取基础 HTML 需 2 RTT；后续 N 个图像请求通过同一 TCP 连续发出，服务器连续响应，只需再加 1 RTT。总时间 = 2 RTT + 1 RTT = **3 RTT** + 传输时间。

## 2.2.3 HTTP 报文格式
### 重要性
★★★★
### 要背的边角料
- **HTTP 报文类型**: 请求报文和响应报文。
- **请求行**: 方法、URL、HTTP 版本。
- **响应状态行**: HTTP 版本、状态码、状态短语。
- **常见请求首部**: `Host` 指明目标主机；`Connection: close` 表示请求关闭非持续连接；`User-Agent` 指明浏览器类型；`Accept-Language` 指明偏好的语言版本。
- **常见响应首部**: `Date` 是服务器产生并发送响应报文的时间；`Last-Modified` 是对象最后修改时间；`Content-Length` 是返回对象的字节数；`Location` 常在 301 响应中指出新 URL。
- **请求方法**: GET (获取), POST (提交), HEAD (只取首部), PUT (上传/替换), DELETE (删除)。
- **实体体 (Entity body)**: GET 请求通常为空；POST 常把表单输入放入实体体；PUT 可向指定 URL 上传对象；DELETE 删除指定 URL 对象；HEAD 类似 GET 但只返回首部。
- **状态码**:
    - `200 OK`: 成功。
    - `301 Moved Permanently`: 永久转移。
    - `304 Not Modified`: 条件 GET 命中，缓存副本仍有效，服务器不返回对象实体。
    - `404 Not Found`: 文档不在服务器上。
    - 400 Bad Request: 一个通用差错代码，指示该请求不能被服务器理解
    - 505 HTTP Version Not Supported：服务器不支持请求报文使用的HTTP版本

### 最可能怎么考
- **填空题**: 哪种方法类似于 GET 但只返回响应首部而不返回对象？（**HEAD**）。
- **应用题**: 给一段 HTTP GET 或响应报文，判断 URL、HTTP 版本、是否持续连接、浏览器类型、状态码、对象长度。
  - **答案套路**：先看请求行或状态行；再看 `Host`、`Connection`、`User-Agent`、`Date`、`Last-Modified`、`Content-Length` 等首部行。注意 `Date` 不是对象最后修改时间，`Last-Modified` 才是对象最后修改时间。

## 2.2.4 Cookie
### 重要性
★★★★
### 要背的边角料
- **4 个组件**: 响应首部 Set-cookie、请求首部 Cookie、浏览器管理文件、后端数据库。
- **作用**: 在无状态的 HTTP 之上建立用户会话层。

### 最可能怎么考
- **简答题**: 描述 Cookie 的完整工作原理（4步模型）。
  - **答案**：
    1. **分配 ID**：服务器生成唯一 ID，并通过 HTTP 响应首部 Set-cookie 发给客户。
    2. **本地存储**：浏览器提取 ID，将其与对应的域名关联，保存在本地的文件中。
    3. **后续携带**：该客户再次向该域名发请求时，浏览器会自动在首部加入 Cookie: ID。
    4. **身份识别**：服务器根据 ID 检索后端数据库，恢复状态，从而在无状态的 HTTP 之上实现了会话层。

## 2.2.5 Web 缓存 (代理服务器)
### 重要性
★★★★
### 要背的边角料
- **优点**: 减少响应时间；减少机构接入链路流量。
- **基本流程**: 浏览器先向 Web 缓存器发送 HTTP 请求；若缓存命中，缓存器直接返回对象；若未命中，缓存器向初始服务器请求对象，收到后保存副本并转发给浏览器。
- **双重角色**: Web 缓存器对浏览器来说是服务器，对初始服务器来说是客户。
- **条件 GET**: `If-Modified-Since: <date>`。
    - 未修改返回 `304 Not Modified`。
    - 若对象已修改，初始服务器返回 `200 OK` 并携带新的对象实体。
    - 构成条件 GET 的两个关键：请求方法是 **GET**；请求首部包含 **If-Modified-Since** 行。

### 最可能怎么考
- **简答题**: Web 缓存器如何减少响应时间和机构接入链路流量？
  - **答案**：当请求对象已缓存在本地时，缓存器可以直接向浏览器返回对象，避免访问远端初始服务器，从而缩短响应时间；同时，命中缓存的请求不再经过机构到因特网的接入链路，减少接入链路流量。
- **计算题/应用题**: 通过命中率计算平均响应时间的改善。
  - **答案解析**：设命中率为 h，本地缓存响应时间为 T_local，源服务器响应时间为 T_remote。
    平均响应时间 = h * T_local + (1 - h) * T_remote。

## 2.2.6 HTTP/2 & HTTP/3
### 重要性
★★★★★
### 要背的边角料
- **HTTP/1.1**: 应用层队首阻塞 (HOL, Head-of-Line blocking)。
- **HTTP/2**: 二进制分帧，解决应用层 HOL，但有 **TCP 级 HOL**。（所以只要前面某个 TCP 段丢失，后面的数据都会被卡住，直到丢失的 TCP 段重传回来。）
- **HTTP/3 (QUIC)**: 基于 **UDP**，彻底解决 HOL 阻塞。

### 最可能怎么考
- **简答题**: 简述 HTTP/1.1、HTTP/2、HTTP/3 分别是如何处理队首阻塞 (HOL) 问题的？
  - **答案**：
    1. **HTTP/1.1**：存在**应用层队首阻塞**。因为请求必须严格按顺序排队响应。
    2. **HTTP/2**：引入**二进制分帧**和流交错，解决了应用层 HOL；但如果 TCP 丢包，会阻塞所有复用在该 TCP 上的流，存在 **TCP级队首阻塞**。
    3. **HTTP/3**：摒弃 TCP，改用基于 UDP 的 **QUIC 协议**，某一流丢包绝不影响其他流，**彻底解决了 HOL 阻塞**。

# 2.3 电子邮件
## 2.3.1 SMTP
### 重要性
★★★
### 要背的边角料
- **电子邮件系统三要素**: 用户代理 (user agent)、邮件服务器 (mail server)、SMTP（简单邮件传输协议）。
- **用户代理功能**: 允许用户阅读、回复、转发、保存和撰写邮件；Outlook、Apple Mail、Gmail 客户端都属于用户代理。
- **邮箱与报文队列**: 接收方邮件服务器维护用户的 **mailbox (邮箱)**；发送方邮件服务器若暂时无法交付，会把邮件放入 **message queue (报文队列)** 并稍后重试。
- **端口**: 25 (基于 TCP)。SMTP 是一种 **推 (Push)** 协议（HTTP是拉协议）。
- **基本命令流程**: `HELO` 建立会话，`MAIL FROM` 指明发送方，`RCPT TO` 指明接收方，`DATA` 开始传输邮件内容，`QUIT` 结束会话。
- **常见响应码**: `220` 表示服务就绪，`250 OK` 表示命令成功，`354` 表示可以开始输入邮件数据。
- **致命限制**: SMTP 要求报文体必须是 **7 比特 ASCII (美国信息交换标准代码) 编码**。
- **MIME (多用途因特网邮件扩展)**: 为了发送非 ASCII 文本（如中文）或二进制附件，必须使用 MIME 将其编码为 7 位 ASCII。
- **结束标志**: 结束报文体输入的标志是单独一行的句点：CRLF . CRLF。

### 最可能怎么考
- **填空题**: SMTP 是一种 **推 (Push)** 协议，而 HTTP 是一种拉协议。
- **简答题**: 简述 SMTP 发送邮件时的基本命令顺序。
  - **答案**：客户与邮件服务器建立 TCP 连接后，通常先用 `HELO` 标识自己，再用 `MAIL FROM` 指明发送方，用 `RCPT TO` 指明接收方，用 `DATA` 发送邮件内容，最后用 `QUIT` 结束会话。
- **简答题**: 为什么在发送带有图片附件的电子邮件时需要使用 MIME 协议？
  - **答案**：因为传统的 SMTP 协议要求邮件的所有内容都必须采用 **7 比特 ASCII 格式** 编码。直接发送二进制的图片数据会引发错误，因此必须使用 **MIME 协议** 将二进制多媒体数据转换为符合 SMTP 要求的文本格式后才能传输。

## 2.3.2 邮件报文格式
### 重要性
★★
### 要背的边角料
- **标准**: RFC (征求意见文档) 5322。
- **结构**: 首部行、空白行、报文体。常见首部包括 `From:`、`To:` 和 `Subject:`。
- **区分**: 邮件报文首部不同于 SMTP 会话中的 `MAIL FROM`、`RCPT TO` 等命令。`MAIL FROM` 属于 SMTP 信封命令，用于邮件传输过程；`From:` 是邮件报文自身的首部行，是收件人看到的邮件内容的一部分。
- **`Received:` 首部行**: 邮件每经过一个邮件服务器，通常会在首部顶部追加一个 `Received:` 行；最上面通常是最后添加的，最下面通常更接近发送源头。Gmail 等 Web 邮件通常只暴露服务商服务器信息，不一定暴露用户真实主机 IP。

### 最可能怎么考
- **填空题**: 邮件报文首部与报文体之间使用 **空白行** 分隔。
- **简答题**: SMTP 中的 `MAIL FROM` 和邮件报文自身的 `From:` 有什么不同？
  - **答案**：`MAIL FROM` 是 SMTP 会话中的命令，属于传输邮件时使用的信封信息；`From:` 是邮件报文首部字段，属于邮件内容本身，用来表示邮件显示的发送者。
- **简答题**: 邮件首部中的 `Received:` 行能说明什么？
  - **答案**：`Received:` 行记录邮件经过的邮件服务器或中继路径，通常包含 `from`、`by`、`with`、时间戳等信息；每经过一个邮件服务器通常会新增一行，因此可用于粗略分析邮件转发路径，但不一定能确定发送者真实主机 IP。

## 2.3.3 邮件访问协议
### 重要性
★★★
### 要背的边角料
- **SMTP 的角色**: SMTP 是**推协议**，适合把邮件从发送方推到邮件服务器、再从发送方邮件服务器推到接收方邮件服务器。
- **两步中继原因**: Alice 的用户代理通常先把邮件交给自己的邮件服务器；若 Bob 的邮件服务器暂时不可达，Alice 的邮件服务器可以持续重试发送。
- **收信不能用 SMTP**: 接收方从自己的邮件服务器取回邮件是**拉操作**，因此不能用 SMTP 完成。
- **第八版重点的两种取信方式**:
    - **HTTP**: 基于 Web 的电子邮件或手机客户端常用，例如 Gmail。
    - **IMAP**: 因特网邮件访问协议，RFC 3501，常见于 Outlook 等邮件客户端。
- **共同能力**: HTTP 和 IMAP 都支持用户管理服务器邮箱中的文件夹，例如移动邮件、删除邮件、标记重要邮件。

### 最可能怎么考
- **简答题**: 为什么 Bob 的用户代理不能用 SMTP 从邮件服务器取回邮件？第八版给出的常用取信方式有哪些？
  - **答案**：SMTP 是一种**推协议**，用于发送方把邮件推送到邮件服务器，以及邮件服务器之间中继邮件；而用户从自己的邮件服务器取回邮件是**拉操作**，所以不能用 SMTP。第八版中常见取信方式有两种：一种是基于 Web 邮件或手机客户端使用 **HTTP**；另一种是邮件客户端使用 **IMAP**。HTTP 和 IMAP 都支持管理服务器上的邮箱文件夹。
- **应用题**: Alice 用 Gmail/Hotmail 这类 Web 邮箱给 Bob 发邮件，Bob 用 IMAP 客户端收信，报文经过哪些应用层协议？
  - **答案**：Alice 的用户代理先用 **HTTP** 将邮件提交到 Alice 的邮件服务器；Alice 的邮件服务器再用 **SMTP** 将邮件中继到 Bob 的邮件服务器；Bob 的用户代理最后用 **IMAP** 从 Bob 的邮件服务器取回并管理邮件。

# 2.4 DNS：因特网的目录服务
## 2.4.1 DNS 提供的服务
### 重要性
★★★★
### 要背的边角料
- **DNS（域名系统） 本质**: DNS 是由分层 DNS 服务器实现的**分布式数据库**，同时也是运行在端系统之间的应用层协议。
- **服务**: 域名解析、主机别名、邮件服务器别名、负载分配。
- **主机别名**: DNS 可用 **CNAME (规范名称)** 记录把别名映射到规范主机名。
- **邮件服务器别名**: DNS 用 **MX (邮件交换)** 记录返回邮件服务器的规范主机名。
- **负载分配 (Load Distribution)**: 繁忙站点可把同一个规范主机名对应到一组冗余服务器 IP 地址；DNS 回答时返回整个 IP 地址集合，并在不同回答中循环这些地址的顺序。由于客户通常连接排在最前面的 IP，负载就被分散到多台服务器上。
- **协议与端口陷阱 (🚨 408 必考)**: 
    - 绝大多数普通客户端查询使用 **UDP，端口 53**（避免 TCP 握手开销）。
    - 当进行 **区域传送 (Zone transfer，主从同步)**，或者响应报文 **超过 512 字节** 时，必须使用 **TCP，端口 53**。

### 最可能怎么考
- **填空题**: DNS 通常运行在 **UDP** 协议之上，使用端口号 **53**。
- **填空题**: DNS 中用于主机别名到规范主机名映射的资源记录类型是 **CNAME**；用于邮件服务器别名的资源记录类型是 **MX**。
- **填空题**: DNS 通过返回同一规范主机名对应的 **IP 地址集合**，并循环改变地址顺序，实现服务器间的 **负载分配**。
- **简答题**: DNS 如何帮助繁忙站点做负载分配？
  - **答案**：繁忙站点通常有多台冗余 Web 服务器，每台服务器有不同 IP，但共享同一个规范主机名。DNS 数据库保存这个 IP 地址集合；当客户查询该主机名时，DNS 返回整个地址集合，并在不同回答中轮换地址顺序。客户通常访问排在最前面的 IP，因此请求会被分散到多台服务器。
- **简答题**: DNS 协议在什么时候会使用 UDP？什么时候会使用 TCP？
  - **答案**：通常的域名解析查询为了降低时延开销，使用的是 **UDP** 协议；但是，当 DNS 响应报文长度超过 512 字节，或者主从 DNS 服务器之间进行大规模的“区域传送 (Zone Transfer)”时，为了保证数据可靠，使用的是 **TCP** 协议。

## 2.4.2 DNS 层次结构与查询
### 重要性
★★★★★
### 要背的边角料
- **DNS 服务器层次**:
    - **根 DNS 服务器**: 返回负责顶级域的 TLD (顶级域) 服务器信息。
    - **TLD 服务器**: 负责 `.com`、`.org`、国家顶级域等，返回权威 DNS 服务器信息。
    - **权威 DNS 服务器**: 保存目标主机名对应的最终记录。
    - **本地 DNS 服务器**: 主机首先查询的 DNS 服务器，通常由 ISP 或机构提供。
- **查询方式**: **递归**（我帮你问）与 **迭代**（去问他）。
- **根 DNS 服务器**: 共有 **13 个逻辑根服务器标识 (A-M)**，实际通过任播部署了大量服务器实例。
- **DNS 缓存**: DNS 服务器可缓存查询结果，减少查询时延和上层服务器负载；缓存记录受 **TTL (生存时间)** 限制。
- **缓存过期/陈旧**: 缓存记录不是永久的；在 TTL 到期前，缓存记录可能暂时与权威记录不一致，TTL 到期后应被丢弃或重新查询。

### 最可能怎么考
- **简答题**: 本地 DNS、根 DNS、TLD DNS、权威 DNS 在域名解析中分别起什么作用？
  - **答案**：主机先把请求交给本地 DNS；本地 DNS 若无缓存，则向根 DNS 查询，根 DNS 返回对应 TLD 服务器；TLD 服务器返回目标域的权威 DNS 服务器；权威 DNS 服务器返回最终资源记录，本地 DNS 再把结果交还给主机。
- **应用题/简答题**: 描述客户端使用“迭代查询”方式获取一个域名的完整解析过程。
  - **答案**：
    1. 本地主机向**本地 DNS 服务器**发送递归请求。
    2. 本地 DNS 向**根服务器**发出请求，根返回负责该域的 **TLD 服务器** IP。
    3. 本地 DNS 向 **TLD 服务器**发出请求，TLD 返回负责该主机的**权威 DNS 服务器** IP。
    4. 本地 DNS 向**权威服务器**发出请求，最终获得目标主机的 IP 并返回给本地主机。

## 2.4.3 DNS 记录 (RR, 资源记录)
### 重要性
★★★★★
### 要背的边角料
- **RR 四元组**: 资源记录格式为 **(Name, Value, Type, TTL)**；TTL 决定该记录应从缓存中删除的时间。
- **Type=A**: 主机名 -> IP。
- **Type=NS**: 域 -> 权威服务器名。
- **Type=CNAME**: 别名 -> 规范主机名。
- **Type=MX**: 别名 -> 邮件服务器名。
- **DNS 报文区域**: 问题区域放查询的问题；回答区域放查询结果 RR；权威区域放权威服务器相关记录；附加区域放有帮助的补充记录。
- **DNS 报文首部**: 前 12 字节是首部区域；**标识符 (Identifier)** 用于匹配查询和回答，会从查询报文复制到回答报文中。
- **DNS 报文标志位**: 常见标志包括查询/回答、权威回答、希望递归、递归可用；首部还包含四个区域中各类记录数量。
- **附加区域典型用途**: MX 查询的回答区域可返回邮件服务器规范主机名，附加区域可顺带给出该规范主机名的 A 记录，减少下一次查询。
- **在 DNS 数据库中插入记录**: 新域名先到**注册登记机构 (Registrar)** 注册；注册时提供基本/辅助权威 DNS 服务器的名字和 IP 地址。注册登记机构负责把该域的 **NS 记录** 和对应权威 DNS 服务器的 **A 记录** 插入 TLD 服务器；域名拥有者还要在自己的权威 DNS 服务器中加入 Web 服务器的 A 记录、邮件服务器的 MX 记录等。
- **DNS 动态更新**: 传统 DNS 记录多由管理员静态配置；DNS 的 UPDATE 机制允许通过 DNS 报文动态增加或删除数据库记录。

### 最可能怎么考
- **简答题**: 注册新域时必须向顶级域服务器插入哪两类核心资源记录？
  - **答案解析**：当注册新域名（如 network.com）时，至少需提供两条记录：
    1. 提供权威名称服务器名称的 **NS 记录**：(network.com, dns1.network.com, NS)。
    2. 提供该名称服务器 IP 的 **A 记录**：(dns1.network.com, 212.2.2.2, A)。
- **填空题**: DNS 回答报文中用于存放资源记录答案的区域称为 **回答区域**；用于存放额外有帮助记录的区域称为 **附加区域**。
- **填空题**: DNS 查询报文和回答报文通过 **标识符 (Identifier)** 匹配。
- **简答题**: 一个新公司要让别人访问它的 Web 站点并给它发邮件，DNS 记录需要怎样插入？
  - **答案**：先向注册登记机构注册域名，并提供本公司的权威 DNS 服务器名字和 IP。注册登记机构把该域的 NS 记录和权威 DNS 服务器的 A 记录放入对应 TLD 服务器；公司还要在自己的权威 DNS 服务器中加入 Web 服务器的 A 记录和邮件服务器的 MX 记录。

# 2.5 P2P 文件分发
## 2.5.1 P2P 体系结构的自扩展性
### 重要性
★★★★
### 要背的边角料
- **BitTorrent (比特洪流)**: 最稀缺优先、一报还一报 (Tit-for-tat)。
- **覆盖网络 (Overlay Network)**: P2P 应用层形成的逻辑网络；节点是对等方，边是对等方之间的 TCP 连接，不包括底层路由器。
- **P2P 分发时间公式变量定义表**:
| 变量 | 含义 |
| :--- | :--- |
| **$F$** | 分发的文件大小 (bits) |
| **$u_s$** | 服务器的上载速率 (bps) |
| **$d_i$** | 第 $i$ 个对等方的下载速率 (bps) |
| **$u_i$** | 第 $i$ 个对等方的上载速率 (bps) |
| **$N$** | 对等方的数量 |
| **$d_{min}$** | 最小的下载速率 $\min\{d_i\}$ |

### 最可能怎么考
- **简答题**: 什么是覆盖网络？它包括底层路由器吗？
  - **答案**：覆盖网络是运行在底层因特网之上的应用层逻辑网络。在 P2P 文件分发中，覆盖网络的节点是参与分发的对等方，边是对等方之间建立的 TCP 连接；它不把底层物理链路和路由器当作覆盖网络节点。
- **计算题/应用题**: 比较 C/S 与 P2P 分发文件的最小时间理论下限推导。
  - **答案解析 (高频大题)**：
    1. **C/S 架构**：分发时间 D_cs = max{ (N*F)/u_s, F/d_min }。
    2. **P2P 架构**：分发时间 D_p2p = max{ F/u_s, F/d_min, (N*F)/(u_s + sum(u_i)) }。
       结论：随 N 增大，C/S 时间呈线性爆炸增长，而 P2P 因总上载速率增加趋于平缓。

## 2.5.2 BitTorrent
### 重要性
★★★
### 要背的边角料
- **术语**: 洪流 (Torrent)、块 (Chunk)、追踪器 (Tracker)。
- **Torrent**: 参与同一个文件分发的一组对等方集合；文件被切分为许多 chunk 在对等方之间交换。
- **Tracker**: 帮助新加入的对等方获得其他对等方列表；真正的数据块通常在对等方之间直接传输。
- **块请求策略**: 最稀缺优先 (Rarest First)。
- **激励机制**: 一报还一报 (Tit-for-tat)，优先疏通向自己上传速率较高的对等方。
- **一报还一报不是严格同一瞬间互传**: Bob 不一定必须在 Alice 上传的同一个 30 秒间隔内立刻回传，但长期不给别人上传会失去优先服务机会。
- **乐观疏通 (Optimistic Unchoking)**: 周期性随机选择一个对等方上传，给新加入者或暂时没有贡献者获得第一个块的机会。

### 最可能怎么考
- **填空题**: BitTorrent 优先请求邻居中最少副本的块，这称为 **最稀缺优先**。
- **简答题**: 新加入的 BitTorrent 对等方没有任何块，为什么仍可能获得第一个块？
  - **答案**：虽然它暂时无法通过一报还一报回报别人，但 BitTorrent 存在**乐观疏通**机制，某些对等方会周期性随机选择新的上传对象，因此新加入者有机会先获得块；之后它就可以把已有块上传给别人，进入一报还一报机制。
- **简答题**: Alice 在某个 30 秒间隔向 Bob 上传块，Bob 是否必须在同一个 30 秒间隔立刻回传？
  - **答案**：不一定。BitTorrent 的一报还一报是长期激励机制，不要求严格在同一个 30 秒窗口立刻互传；但若 Bob 长期不给别人上传，就难以成为其他对等方的优先上传对象。

# 2.6 视频流和内容分发网 (CDN)
## 2.6.1 视频流
### 重要性
★★★
### 要背的边角料
- **核心要求**: 平均端到端吞吐量必须满足视频播放速率，且应控制启动时延和卡顿。
- **预取缓冲**: 客户端先缓存一部分视频数据，再开始播放，以吸收网络速率波动。

### 最可能怎么考
- **简答题**: 为什么视频流播放前需要预取缓冲？
  - **答案**：视频播放要求持续、稳定地消耗数据，而网络吞吐量会随时间波动。客户端先预取并缓存一部分视频字节，等缓存超过门限后再开始播放；播放时一边从缓存中取帧解码显示，一边继续下载后续视频。这样短时间的带宽下降不会立刻造成卡顿，只要平均端到端吞吐量不低于视频编码比特率，就能维持连续播放。

## 2.6.2 DASH (经 HTTP 的动态适应性流)
### 重要性
★★★
### 要背的边角料
- **DASH (经 HTTP 的动态适应性流)**: 客户端根据当前测量的**可用带宽**动态请求不同码率的视频块。
- **告示文件 (Manifest file)**: 服务器提供的一个索引文件，包含了不同版本视频的 URL 及其对应的比特率。客户端根据它去请求视频块。
- **分块与多版本**: 视频被切成多个块，每个块可有不同码率版本；客户端逐块选择合适版本请求。

### 最可能怎么考
- **简答题**: 在 DASH 协议中，客户端是如何实现码率“动态自适应”的？告示文件起了什么作用？
  - **答案**：客户端通过持续测量其收到的 HTTP 响应速率来估算当前可用带宽；然后它会查阅**告示文件**（列出视频块的可用版本、码率及 URL），根据带宽动态决定请求哪一个码率的块。
- **应用题**: 若一个 DASH 系统有 N 个视频版本和 N 个音频版本，混合存储与分开存储分别需要多少个文件？
  - **答案**：如果每种音频和视频组合都混合成一个文件，需要 $N^2$ 个文件；如果音频和视频分开存储，分别选择并同步播放，只需要 $2N$ 个文件。

## 2.6.3 CDN 策略
### 重要性
★★★★
### 要背的边角料
- **深入 (Enter Deep)**: 在大量 ISP 中部署小集群。
- **邀请做客 (Bring Home)**: 在少量 IXP 中建设大集群。
- **DNS 重定向**: 许多 CDN 利用 DNS 截获并重定向用户请求，把客户导向某个合适的 CDN 集群或服务器。
- **服务器选择策略**: CDN 要把客户导向合适的副本服务器，常见依据包括地理距离、网络时延、可用带宽、服务器负载、运营成本和 ISP/流量费用。
- **集群选择陷阱**: 最接近的地理位置不一定等于网络路径最近；基于 DNS 的选择常看到的是本地 DNS 服务器位置，而不一定是终端用户真实位置。

### 最可能怎么考
- **简答题**: 描述视频分发中两种 CDN 部署策略（Enter Deep 与 Bring Home）的优缺点。
  - **答案**：
    1. **深入 (Enter Deep)**：在遍及全球的数百个 ISP 中部署小型集群。**优点**：缩短距离，时延极低；**缺点**：管理维护数以千计的集群成本极高。
    2. **邀请做客 (Bring Home)**：在少量的 IXP 处建立大型集群。**优点**：集中化管理、维护成本低；**缺点**：对部分偏远客户时延相对较高。
- **简答题**: CDN 选择副本服务器时，除了网络时延、丢包和带宽外，还可能考虑哪些因素？
  - **答案**：还可能考虑服务器当前负载、内容是否已缓存、运营成本、不同 ISP 之间的流量费用、商业策略以及故障避让。实际选择不一定只选地理位置最近的服务器。

## 2.6.4 学习案例：Netflix 和 YouTube
### 重要性
★★
### 要背的边角料
- **本节地位**: 案例应用，说明 DASH、CDN、缓存和 HTTP 流在真实系统中的使用；优先级低于 2.6.2 和 2.6.3。
- **Netflix**:
    - 亚马逊云负责用户注册、登录、计费、目录、推荐、内容摄取和内容处理。
    - 自建 CDN / Open Connect 负责视频分发。
    - 内容通常在非高峰时段**推送**到 CDN，属于推缓存。
    - 客户端获得 CDN 服务器 IP 和不同版本 URL 后，按吞吐量请求不同质量的视频块，体现 DASH 思想。
- **YouTube**:
    - 依托 Google 数据中心和专用 CDN 分发视频。
    - 上传视频会被转码成多个不同比特率版本。
    - 使用 HTTP 流和 HTTP 字节范围请求，减少跳转、暂停或提前退出造成的带宽浪费。
    - 可用 DNS 重定向把客户导向合适集群，通常考虑 RTT，也可能考虑负载均衡。
### 最可能怎么考
- **填空题**: Netflix 视频分发主要依赖自己的专用 CDN，即 **Open Connect**；YouTube 依托 **Google 数据中心和 CDN**。
- **简答题**: Netflix 架构中亚马逊云和自建 CDN 分别负责什么？
  - **答案**：亚马逊云主要负责登录、计费、目录、推荐、内容摄取、转码和把内容上传到 CDN；Netflix 自建 CDN 主要负责把视频内容分发给用户。也就是说，云端偏控制和处理，CDN 偏实际视频交付。
- **简答题**: Netflix 的推缓存和普通 Web 缓存的拉缓存有什么区别？
  - **答案**：推缓存是在非高峰时段预先把内容放入 CDN 服务器；拉缓存是在用户请求未命中时才从源服务器动态取回内容。Netflix 采用推送热门或完整视频库的方式，适合稳定的大规模视频分发。
- **简答题**: YouTube 为什么使用 HTTP 字节范围请求？
  - **答案**：用户可能拖动进度条、提前退出或切换视频。如果服务器一次性持续发送过多后续内容，会浪费带宽和服务器资源。HTTP 字节范围请求允许客户端只请求视频文件中的特定字节范围，便于按需获取片段，也减少提前终止造成的浪费。
- **简答题**: 比较 Netflix 和 YouTube 的视频分发特点。
  - **答案**：Netflix 主要分发专业视频内容，使用亚马逊云处理业务和内容，使用自建 CDN/Open Connect 分发视频，并通过推缓存和类似 DASH 的机制选择视频质量；YouTube 面向大量用户上传视频，依托 Google 数据中心和 CDN，上传后转码为多个码率版本，并通过 HTTP 流、字节范围请求和 DNS 重定向进行分发。

# 2.7 套接字编程
## 2.7.1 UDP 套接字编程
### 重要性
★★★
### 要背的边角料
- **UDP 套接字类型**: `SOCK_DGRAM`，表示 UDP 套接字，而不是 TCP 套接字。
- **目的地址**: UDP 发送方必须为报文附上目的地址，即目的主机 IP 地址 + 目的端口号。
- **源地址**: 源主机 IP 地址 + 源端口号也会附在分组上，但通常由底层操作系统自动完成。
- **客户端端口**: UDP 客户创建套接字时通常不显式指定端口号，而是让操作系统自动分配。
- **服务器端口绑定**: UDP 服务器要用 `bind()` 把端口号和服务器套接字绑定，例如绑定到端口 12000。
- **关键调用**: 客户端用 `sendto()` 发送报文；服务器/客户端用 `recvfrom()` 接收报文并得到发送方地址。
- **服务器返回地址**: UDP 服务器从 `recvfrom()` 得到 `clientAddress`，再用它作为 `sendto()` 的目的地址把响应发回客户。

### 最可能怎么考
- **填空题**: UDP 套接字创建时使用的类型参数是 **`SOCK_DGRAM`**。
- **填空题**: UDP 服务器通过 **`bind()`** 将端口号与套接字绑定。
- **简答题**: UDP 发送方为什么必须指定目的 IP 地址和端口号？
  - **答案**：UDP 是无连接的，发送前不会先建立一条固定连接。因此应用程序发送每个数据报时，都要为该数据报附上目的地址，目的地址由目的主机 IP 地址和目的端口号组成，这样网络才能把数据报送到正确主机上的正确进程。
- **简答题**: UDP 服务器收到客户报文后，如何知道应该把响应发回哪里？
  - **答案**：服务器调用 `recvfrom()` 接收报文时，不仅得到报文内容，也得到客户的源地址 `clientAddress`，其中包含客户 IP 地址和端口号。服务器随后用 `sendto()` 把响应发送到这个客户地址。

## 2.7.2 TCP 套接字：欢迎套接字与连接套接字
### 重要性
★★★
### 要背的边角料
- **TCP 套接字类型**: `SOCK_STREAM`，表示 TCP 套接字。
- **TCP 面向连接**: 客户和服务器发送数据前要先握手并建立 TCP 连接。
- **欢迎套接字**: 服务器监听客户连接请求。
- **连接套接字**: 服务器接受连接后创建，用于与特定客户交换数据。
- **关键调用**: 客户端用 `connect()` 发起连接；服务器用 `listen()` 监听，用 `accept()` 接受连接。
- **套接字数量**: UDP 服务器通常只需一个套接字；TCP 服务器至少有一个欢迎套接字，每接受一个客户连接还会创建一个连接套接字。
- **字节流**: TCP 把数据处理为可靠、有序的字节流，不保证一次 `send` 正好对应一次 `recv`。
- **启动顺序**: TCP 客户连接前服务器必须已在监听；UDP 客户可先发送，但若服务器未运行，请求通常无法得到响应。

### 最可能怎么考
- **填空题**: TCP 套接字创建时使用的类型参数是 **`SOCK_STREAM`**。
- **填空题**: TCP 服务器通过 `accept()` 创建与特定客户通信的 **连接套接字**。
- **简答题**: 为什么 UDP 服务器通常只需一个套接字，而 TCP 服务器需要欢迎套接字和连接套接字？
  - **答案**：UDP 无连接，服务器可用同一个套接字接收来自不同客户的数据报，并根据报文中的源地址回复；TCP 面向连接，欢迎套接字只负责监听和接受新的连接请求，`accept()` 后创建的连接套接字才负责与特定客户进行可靠字节流通信。
- **简答题**: TCP 服务器为什么必须先于客户运行？UDP 客户为什么可以先运行？
  - **答案**：TCP 客户需要向服务器的欢迎套接字发起连接，若服务器未监听，连接无法建立；UDP 没有连接建立过程，客户可以先把数据报发出，但如果服务器没有运行，该数据报通常无人接收，也不会得到应用层响应。
- **简答题**: TCP 面向字节流与 UDP 数据报方式有什么区别？
  - **答案**：TCP 在连接两端提供可靠、有序的字节流服务，应用层看到的是连续字节序列，不保证一次读取正好对应一次写入；UDP 按独立数据报发送和接收，每个数据报都携带目的地址，接收方可按数据报获得报文内容和源地址。

# 第 2 章考场小题：题目与解答

## 题 1：HTTP 多对象下载需要多少 RTT？

**题目**：一个 Web 页面包含 1 个基础 HTML 文件和 7 个小图片对象。忽略对象传输时间，分别说明非持续 HTTP、持续非流水线 HTTP、持续流水线 HTTP 大约需要多少 RTT。

**解答**：

- 基础 HTML 通常需要 `2RTT`：`1RTT` 建 TCP，`1RTT` 请求并收到 HTML 响应开头。
- **非持续 HTTP**：每个对象都要单独建立 TCP 连接，8 个对象约为 `8 × 2RTT = 16RTT`。若浏览器允许并行 TCP，小图片可并行，RTT 数会减少。
- **持续非流水线 HTTP**：只建一次 TCP，基础 HTML `2RTT`，后面 7 个图片每个约 `1RTT`，共约 `9RTT`。
- **持续流水线 HTTP**：基础 HTML `2RTT`，发现图片后连续发送 7 个请求，通常再加 `1RTT`，共约 `3RTT`。

**考场写法**：先数对象数，再判断连接方式；非持续按每对象 `2RTT`，持续连接少建 TCP，流水线能把多个请求合并等待。

## 题 2：DNS 查询 + HTTP 访问总时间怎么算？

**题目**：主机访问某网站，若本地 DNS 没有缓存，需要依次查询根 DNS、TLD DNS、权威 DNS，然后访问 Web 服务器。忽略传输时间，求总时延应如何计算。

**解答**：

- DNS 无缓存时，先加 DNS 查询 RTT：本地 DNS 递归，外部服务器迭代，常见过程是根 -> TLD -> 权威。
- 得到 IP 后再访问 Web：TCP 建连 `1RTT`，HTTP 请求/响应 `1RTT`。
- 所以总时间可写为：

$$
T = T_{DNS} + 2RTT_{HTTP}
$$

- 若题目给每次 DNS 查询也是 1 个 RTT，且需要查根、TLD、权威 3 次，则：

$$
T = 3RTT_{DNS} + 2RTT_{HTTP}
$$

**考场写法**：先判断 DNS 是否有缓存；有缓存只算 HTTP，没缓存先算 DNS 再算 TCP + HTTP。

## 题 3：C/S 与 P2P 文件分发时间公式是什么？

**题目**：服务器向 N 个用户分发大小为 F 的文件，服务器上传速率为 `u_s`，最慢客户端下载速率为 `d_min`，各 peer 上传速率为 `u_i`。写出 C/S 和 P2P 的最小分发时间下界。

**解答**：

客户-服务器模式：

$$
D_{CS}=\max\left\{\frac{NF}{u_s},\frac{F}{d_{min}}\right\}
$$

P2P 模式：

$$
D_{P2P}=\max\left\{\frac{F}{u_s},\frac{F}{d_{min}},\frac{NF}{u_s+\sum_{i=1}^{N}u_i}\right\}
$$

**解释**：

- C/S 中服务器至少上传 N 份文件，最慢客户端至少下载一整份文件。
- P2P 中服务器至少注入一整份文件，最慢 peer 至少下载一整份文件，全系统上传能力要支撑 N 份文件扩散。
- P2P 具有自扩展性，因为 peer 下载后也能上传。

## 题 4：邮件协议路径和 MIME 怎么答？

**题目**：用户 A 给用户 B 发送带图片附件的邮件，说明发送和接收过程可能使用哪些协议，并解释为什么需要 MIME。

**解答**：

- 用户代理把邮件交给发送方邮件服务器：通常用 **SMTP**。
- 发送方邮件服务器把邮件转发给接收方邮件服务器：用 **SMTP**。
- 用户 B 从接收方邮件服务器取信：可用 **POP3 / IMAP / HTTP**，取决于是邮件客户端还是 Web 邮箱。
- 传统 SMTP 主要传 7 比特 ASCII 文本，不能直接传图片、音频、附件、中文等二进制或非 ASCII 内容。
- **MIME** 用来扩展邮件内容格式，把非 ASCII 文本和二进制附件编码成适合 SMTP 传输的形式。

**考场写法**：SMTP 负责邮件传输，MIME 负责内容表示和编码扩展；MIME 不是替代 SMTP。
