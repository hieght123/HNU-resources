# 第 3 章 运输层：进程间逻辑通信

### 重要性
★★★★★
运输层把网络层的主机间交付扩展为进程间交付；其中 TCP (传输控制协议) 进一步提供可靠数据传输、流量控制和拥塞控制。

---

# 3.1 运输层服务
## 3.1.1 运输层和网络层的关系
### 重要性
★★★
### 要背的边角料
- **逻辑通信**：运输层提供**进程间**的逻辑通信；网络层提供**主机间**的逻辑通信。
- **端系统限制**：运输层协议只在**端系统**中实现。
- **服务受限与可增强**：运输层受网络层服务模型限制，不能凭空保证网络层没有提供的带宽/时延；但可以在不可靠网络层之上通过确认、重传、序号等机制提供可靠性，也可以通过加密提供机密性。

### 最可能怎么考
- **填空题**：网络层负责主机间通信，而运输层负责 **进程** 间通信。
- **简答题**：简述运输层与网络层服务的本质区别。
  - **答案**：网络层提供了**主机之间**的逻辑通信，它试图将分组从源主机交付给目的主机，但这是尽力而为的服务，不可靠；运输层在网络层之上，提供了运行在不同主机上的**进程之间**的逻辑通信，并且可以通过 TCP 等协议在不可靠的网络层之上建立可靠的数据传输服务。

## 3.1.2 因特网运输层概述
### 重要性
★★★
### 要背的边角料
- **TCP/UDP 服务模型**：
  - **答案**：
    1. **TCP**：面向连接、提供可靠传输、流量控制和拥塞控制。
    2. **UDP**：无连接、不可靠交付、首部开销小（仅8字节）、无拥塞控制（发送速率不被网络拥塞限制）；它只在 IP 之上补充进程间交付、多路复用/分解和差错检测这类基本服务。
- **运输层分组术语**：本书把 TCP 和 UDP 的运输层分组统称为**报文段 (segment)**；
- **简答题**：应用程序为什么可能选择 UDP 而不是 TCP？
  - **答案**：UDP 无连接、首部小、没有拥塞控制带来的发送速率限制，应用可以更直接地控制发送时机和可靠性机制，适合 DNS (域名系统)、实时音视频、游戏等对低时延和控制权更敏感的场景；如果需要可靠性，也可以在应用层自己实现确认、重传等机制。

# 3.2 多路复用与多路分解
### 重要性
★★★★
### 要背的边角料
- **套接字 (socket)**：进程和网络之间的数据门户；运输层实际把报文段数据交付给套接字，再由套接字交给进程。
- **多路分解 (demultiplexing)**：接收端运输层检查报文段字段，将数据定向到正确的套接字。
- **多路复用 (multiplexing)**：发送端运输层从不同套接字收集数据块，为每个数据块封装首部后生成报文段并交给网络层。
- **复用/分解两个要求**：套接字有唯一标识符；每个报文段有特殊字段指示要交付到哪个套接字。
- **端口字段**：运输层报文段中用于分解的核心字段是**源端口号字段**和**目的端口号字段**。
- **最小分解协议**：若网络层只能把报文段交给目的主机，运输层至少要在首部加入目的端口号才能交付给正确进程；若要回包，还要携带源端口号作为返回地址的一部分。
- **UDP 套接字标识**：**二元组** (目的 IP, 目的端口)。
- **TCP 套接字标识**：**四元组** (源 IP, 源端口, 目的 IP, 目的端口)。
- **端口号范围**：端口号是 **16 比特**，范围 **0 到 65535**；**0 到 1023** 是周知端口号。
- **周知端口例子**：HTTP (超文本传输协议) 常用端口 **80**，FTP (文件传输协议) 常用端口 **21**。
- **临时端口**：客户端通常由运输层自动分配 **1024 到 65535** 中未被占用的端口；服务器通常绑定固定端口。
- **bind()**：服务器程序可用 `bind()` 将套接字关联到一个特定端口号。
- **服务器回包端口**：A 发往 B 的 TCP 报文若源端口为 x、目的端口为 y，则 B 回 A 时源端口为 y、目的端口为 x。
- **UDP 分解补充**：虽然 UDP 套接字按目的 IP 和目的端口分解，但报文段中仍携带源 IP 和源端口，应用可据此区分不同发送方并回包。
- **欢迎套接字/连接套接字**：TCP 服务器用欢迎套接字监听端口；收到连接请求后，为该连接创建由四元组标识的连接套接字。
- **Web 服务器与 TCP**：初始连接建立报文段和承载 HTTP 请求的报文段目的端口都可为 80；服务器靠源 IP、源端口等四元组区分不同 TCP 连接。
- **端口扫描 (port scanning)**：端口扫描器用于判断哪些端口打开；教材典型工具是 **nmap**，可返回打开、关闭或不可达端口列表。

### 最可能怎么考
- **填空题**：将运输层报文段中的数据交付到正确套接字的工作称为 **多路分解**；从不同套接字收集数据并封装成报文段称为 **多路复用**。
- **逻辑判断**：两台不同主机的 UDP 报文发往同一服务器端口，会进入同一个 Socket 吗？（是）。
- **逻辑判断**：两台不同主机的 TCP 请求发往服务器端口，会进入同一个 Socket 吗？（否）。
- **填空题**：TCP 连接由 **源 IP、源端口、目的 IP、目的端口** 四元组唯一标识。
- **应用题**：Web 服务器端口 80 同时服务两台客户主机时，请求是否进入同一个连接套接字？
  - **答案**：不是。服务器的欢迎套接字监听 80 端口；每个 TCP 连接由四元组唯一标识，因此来自不同客户主机或不同源端口的连接会对应不同连接套接字，但这些连接套接字的本地端口都可以是 80。

# 3.3 无连接运输：UDP
## 3.3.1 UDP 报文段结构
### 重要性
★★★★
### 要背的边角料
- **UDP 最小主义**：UDP 只在 IP 之上增加多路复用/分解和少量差错检测，发送前没有握手，所以是无连接的。
- **选择 UDP 的理由**：应用可精细控制发送时机；无连接建立时延；无连接状态，服务器可支持更多活跃客户；首部开销小。
- **应用层控制**：应用进程一把数据交给 UDP，UDP 就打包并交给网络层；TCP 可能因拥塞控制、可靠传输或缓存策略延迟发送。
- **无连接状态**：UDP 不维护接收/发送缓存、拥塞控制参数、序号和确认号等连接状态。
- **UDP 拥塞风险**：UDP 没有拥塞控制，若应用不自律，可能造成高丢包并挤压 TCP 会话。
- **UDP 上的可靠性**：应用可以在 UDP 之上自己实现确认、重传等可靠性机制，例如 QUIC (快速 UDP 因特网连接)；但这会把复杂性转移给应用层。
- **常见应用选择**：电子邮件、远程终端访问、Web、文件传输通常运行在 TCP 之上；DNS (域名系统) 常运行在 UDP 之上以避免 TCP 建连时延；SNMP (简单网络管理协议) 常用 UDP；多媒体应用可使用 UDP 或 TCP。
- **多媒体走 TCP 的现实原因**：语音、视频流量有时经 TCP 发送，不一定是因为 TCP 拥塞控制更合适，而是因为许多防火墙或 NAT 会阻挡 UDP，应用常把 TCP 当作可穿透的备选路径。
- **UDP 对数据边界的控制**：UDP 一次从应用进程接收一块数据就形成一个 UDP 报文段；TCP 是字节流，可能按缓存、MSS、拥塞控制等因素合并或拆分应用写入的数据。
- **首部长度**：固定 **8 字节**。
- **四个字段**：源端口、目的端口、长度、检验和，**每个字段 2 字节 = 16 比特**。
- **长度字段**：UDP 长度包括首部和应用数据。
- **检验和范围**：UDP 检验和覆盖 UDP 首部、应用数据以及伪首部中的部分 IP 层信息。

### 最可能怎么考
- **填空题**：UDP 首部由 **4** 个字段组成，总共 **8** 字节。

## 3.3.2 UDP 检验和 (Checksum)
### 重要性
★★★
### 要背的边角料
- **计算逻辑**：16 比特字求和，**溢出回卷**，最后求**反码**。
- **端到端原则**：即使链路层有校验，运输层仍需检验以应对路由器内存错误。
- **检错局限**：检验和匹配只能说明“未检测出差错”，不能绝对保证没有比特差错。
- **差错恢复边界**：UDP 只提供差错检测，不负责差错恢复；实现上可能丢弃受损报文段，也可能把受损报文段交给应用程序并给出警告。

### 最可能怎么考
- **计算题**：求给定二进制串的 UDP 检验和。
  - **答案解析**：把数据按 16 比特分组相加，若最高位产生进位则**回卷**加到低位，最后对结果逐位取反。例：`0000000000000001 + 0000000000000010 = 0000000000000011`，所以检验和为 `1111111111111100`。接收方把所有 16 比特字和检验和再相加，若结果全为 1，则认为未检测出差错。
- **判断题**：UDP 检验和正确就能绝对确认没有任何比特差错。（错误：只能说明没有检测出差错。）

# 3.4 可靠数据传输原理 (rdt)
## 3.4.1 构造可靠数据传输协议 (rdt 1.0 - 3.0)
### 重要性
★★★★★
### 要背的边角料
- **可靠信道抽象**：不丢失、不损坏、按发送顺序交付。
- **rdt 接口名**：上层调用 `rdt_send()` 交付待发送数据；接收端从下层收到分组时触发 `rdt_rcv()`；向上交付数据用 `deliver_data()`。
- **单向数据传输假设**：教材构造 rdt 时先只讨论数据从发送方到接收方的单向传输，但控制分组会双向流动。
- **ARQ（自动重传） 三要素**：差错检测、接收方反馈、重传。
- **表 3-1 机制清单**：检验和用于检测比特差错；定时器用于超时重传；序号用于区分新分组和重传；ACK/NAK (肯定确认/否定确认) 用于反馈接收结果；窗口和流水线用于提升利用率。
- **ACK/NAK**：ACK 是肯定确认，NAK 是否定确认；rdt 2.2 用携带序号的 ACK 取消 NAK。
- **ACK 分组序号**：rdt3.0 的 ACK 分组本身不需要另设序号，因为 ACK 字段已经说明它确认的是哪个数据分组；重复 ACK 通常只会导致发送方保持或重发当前分组。
- **只用 NAK 的边界**：若发送方只是偶尔发送数据，只用 NAK 不如 ACK 稳定，因为接收方可能不知道有分组缺失；若连续大量发送且丢包很少，NAK 才可能减少反馈量。
- **FSM**：有限状态机，用状态、事件和动作描述协议行为。
- **FSM 图符号**：状态变迁线上方是事件，下方是动作；`Λ` 表示没有事件或没有动作。
- **rdt 1.0**: 底层完全可靠，不需要反馈。
- **rdt 2.0**: 处理比特差错，引入 ACK/NAK 和重传，但没处理 ACK/NAK 损坏。停等协议
- **rdt 2.1**: 引入序号 (0, 1)。处理 ACK/NAK 损坏。
- **rdt 2.2**: 无 NAK，ACK 携带序号。
- **rdt 3.0**: 引入定时器，处理丢包。重传
- **比特交替协议**: rdt 3.0 的分组序号在 0 和 1 之间交替。
- **序号作用**: 让接收方区分新分组和重传的重复分组。
- **定时器作用**: 发送方在分组或 ACK 丢失时，能通过超时重传恢复。

### 最可能怎么考
- **简答题 0**: rdt 协议为什么需要序号和定时器？
  - **答案**：序号用于区分新数据和重复数据，避免把重传分组错误交付给上层；定时器用于处理数据分组或 ACK 丢失的情况，发送方在等待确认超时后进行重传。
- **简答题 1**: rdt 2.1 中接收方收到重复分组时的两个动作是什么？
  - **答案**：动作 A：丢弃该重复分组。动作 B：重新发送上一个分组的 ACK（告知发送方自己已正确收到）。
- **简答题 2**: rdt 3.0 协议中，若超时时间设置得过短会发生什么？
  - **答案**：这会导致大量不必要的**过早超时 (Premature timeout)**。发送方会在正常的 ACK 到达之前重传分组，不仅浪费了信道带宽，还向网络中注入了大量的重复数据分组。
- **简答题 3**: 如果 RTT (往返时间) 固定且发送方知道 RTT，但分组可能丢失，rdt 3.0 还需要定时器吗？
  - **答案**：需要。即使 RTT 固定，数据分组或 ACK 仍可能丢失；如果没有定时器，发送方可能永远等待一个不会到达的 ACK，协议无法从丢包中恢复。

## 3.4.2 流水线可靠数据传输协议
### 重要性
★★★★
### 要背的边角料
- **停等利用率公式**: $U_{sender} = \frac{L/R}{RTT + L/R}$。
- **流水线利用率公式**: $U_{sender} = \frac{N \times (L/R)}{RTT + L/R}$。
- **停等利用率**: 停等协议一次只能有一个未确认分组，链路长、RTT 大时利用率很低。
- **流水线核心收益**: 允许多个分组未确认时继续发送，从而显著提高信道利用率。
- **流水线代价**: 需要更大的序号范围，并且发送方/接收方可能需要缓存多个分组。
- **差错恢复两类方法**：流水线可靠传输中处理丢失、损坏和过度延时分组的两种基本方法是 **GBN (回退 N 步)** 和 **SR (选择重传)**。

### 最可能怎么考
- **计算题/应用题**: 求使信道利用率达标所需的最小窗口大小 N。
  - **答案解析**：为了使信道利用率 U 达标，发送窗口 N 必须满足公式 $N \ge \frac{U \times (RTT + L/R)}{L/R}$。例如若要求信道满利用，$RTT=99ms$，$L/R=1ms$，则 $N \ge (99+1)/1=100$，窗口至少为 100 个分组。

## 3.4.3 回退 N 步 (GBN)
### 重要性
★★★★
### 要背的边角料
- **窗口变量**: `base` 是最早未确认分组序号，`nextseqnum` 是下一个可用序号。
- **窗口长度**: GBN 中已发送但未确认的许可序号范围可看作长度为 N 的窗口；窗口向前滑动，所以 GBN 是一种**滑动窗口协议**。
- **特性**: **累积确认**、重传所有未确认分组。
- **ACK 含义**: ACK n 表示 n 及以前的分组已经按序收到。
- **窗口外 ACK**: GBN 发送方可能收到已经落在当前窗口外的旧 ACK 或重复 ACK，这是 ACK 在网络中延迟传播、窗口已滑动造成的，不代表协议错误。
- **窗口限制**: $N \le 2^k - 1$。
- **序号范围**: 若序号字段为 k 比特，则序号空间为 $0$ 到 $2^k-1$，按模 $2^k$ 回卷。
- **接收方行为**: 只按序接收，不缓存失序分组；收到失序分组时丢弃并重发最近按序分组的 ACK。
- **定时器**: 通常只为最早未确认分组设置一个定时器。
- **GBN 三类事件**: 发送方主要响应来自上层的数据、收到 ACK、超时事件。

### 最可能怎么考
- **填空题**: GBN 协议中，若发送窗口为 8，序号至少需要 **4** 位。
- **应用题**: GBN 中若分组 2 丢失，而分组 3、4 到达接收方，接收方如何处理？
  - **答案**：接收方不缓存失序分组，会丢弃分组 3、4，并重复发送对最后一个按序收到分组的 ACK；发送方超时后会从丢失分组开始回退重传所有未确认分组。

## 3.4.4 选择重传 (SR)
### 重要性
★★★★
### 要背的边角料
- **特性**: 逐个确认、缓存失序分组。
- **窗口限制**: $W_{send} + W_{recv} \le 2^k$。通常 $W \le 2^{k-1}$。
- **与 GBN 区别**: SR 只重传真正丢失或出错的分组，接收方会缓存失序分组；GBN 会丢弃失序分组并回退重传多个分组。
- **定时器**: 发送方通常为每个未确认分组分别维护定时器。
- **接收方重确认**: SR 接收方收到已经接收过、但序号小于当前窗口基序号的分组时，应重新确认而不是简单忽略，否则发送方窗口可能无法前移。
- **窗口不同步风险**: SR 中发送方和接收方并不总是知道对方窗口位置；序号空间过小或窗口过大时，会无法区分旧分组重传和序号回卷后的新分组。
- **窗口为 1 的退化**: 当发送方窗口和接收方窗口都为 1 时，SR 和 GBN 都退化为比特交替协议；窗口外 ACK 也可能只是旧 ACK 延迟到达。

### 最可能怎么考
- **简答题/逻辑推断**: 解释为什么 SR 协议中窗口大小不能超过序号空间的一半？
  - **答案解析 (408 必考)**：如果序号空间大小为 $2^k$，则 $W_{send} + W_{recv} \le 2^k$。在收发窗口相等时，**$W \le 2^{k-1}$ (即序号空间的一半)**。如果窗口过大（例如序号 0-3，窗口为 3），当 ACK 全部丢失导致发送方重传序号 0 的旧分组时，接收方由于窗口已经滑动，会错误地把这个旧分组当成“新一轮序号回卷后”的新分组接收，导致数据损坏。
- **简答题**: GBN 和 SR 的核心区别是什么？
  - **答案**：GBN 使用累积确认，接收方不缓存失序分组，丢包后发送方会回退重传多个未确认分组；SR 使用逐个确认，接收方缓存失序分组，发送方只选择性重传丢失或出错的分组，因此带宽利用更高但实现更复杂。

# 3.5 面向连接的运输：TCP
## 3.5.1 TCP 连接
### 重要性
★★★★
### 要背的边角料
- **MSS (最大报文段长度)**: 指 TCP 报文段中**应用层数据**的最大长度，不包括 TCP/IP 首部。
- **MTU (最大传输单元)**: 以太网常见 MTU 为 **1500 字节**，因此典型 MSS 为 **1460 字节**。
- **面向连接**: 数据交换前必须先握手建立连接，连接状态只保存在端系统中，中间路由器不维护 TCP 连接状态。
- **点对点**: 一条 TCP 连接只连接一个发送方和一个接收方。
- **全双工**: 连接双方可以同时发送和接收数据。
- **字节流服务**: TCP 把应用数据看成有序字节流，不保留应用层报文边界。
- **缓存**: 两端都有发送缓存和接收缓存。
- TCP连接的组成包括：一台主机上的缓存、变量和与进程连接的套接字，以及另一台主机上的另一组缓存、变量和与进程连接的套接字。
### 最可能怎么考
- **填空题**: TCP 连接是 **点对点**、**全双工**、面向连接的可靠字节流服务。
- **简答题**: TCP 连接的状态保存在哪里？路由器是否维护 TCP 连接状态？
  - **答案**：TCP 连接状态保存在通信两端的端系统中，例如序号、确认号、发送/接收缓存、窗口等；中间路由器只转发 IP 数据报，不维护 TCP 连接状态。

## 3.5.2 TCP 报文段结构
### 重要性
★★★★★
### 要背的边角料
- **TCP 首部一般长度**: 通常 **20 字节**，比 UDP 首部多 **12 字节**。
- **TCP 首部字段清单**: 源端口、目的端口、序号、确认号、首部长度、标志位、接收窗口、检验和、紧急数据指针、选项。
- **端口字段**: 源端口号和目的端口号都为 **16 比特**。
- **序号/确认号字段长度**: TCP 序号字段和确认号字段各 **32 比特**。
- **序号 (Seq)**: 报文段首字节的字节流编号（不是报文段的编号）。
- **确认号 (Ack)**: **期望从另一方收到的下一个字节的序号**。TCP 提供**累积确认**。
- **序号增长陷阱**：后继 TCP 报文段的序号不是固定加 1，而是加上前一个报文段携带的数据字节数；序号描述本方向字节流，确认号描述反方向“下一个期望字节”。
- **同段 Seq/Ack 独立**：同一个 TCP 报文段中的序号和确认号服务于两个方向的数据流，不能因为本段 `seq=m`、数据长度为 n，就推出本段 `ack=m+n`。
- 408 头部大小陷阱: TCP 首部最小 **20 字节**。首部中的**首部长度字段 (Data Offset)** 占 4 位，其单位是 **4 字节**（例如该字段为 5，则实际长度为 5 * 4 = 20 字节）。
- **接收窗口字段**: **16 比特**，用于流量控制，表示接收方愿意接收的字节数量。
- **TCP 检验和字段**: **16 比特**，用于检测 TCP 报文段中的比特差错。
- **标志位**: 常考 SYN (同步), FIN (结束), ACK (确认), RST (复位)；完整图中还包括 CWR (拥塞窗口已减小), ECE (显式拥塞回显), PSH (推送), URG (紧急)。ECE/CWR 用于 ECN (显式拥塞通告)，PSH/URG 实践中很少用。
- **ACK 标志位**: ACK 比特用于指示确认号字段中的值是有效的。
- **RST/SYN/FIN**: RST、SYN、FIN 用于连接建立和拆除相关控制。
- **紧急数据指针**: URG 置位时，**16 比特**紧急数据指针指出紧急数据最后一个字节的位置；实践中很少使用。
- **接收方 ACK 行为**: TCP 使用累积确认；若中间字节缺失，即使后续字节到达，也会重复确认“仍然期待的下一个字节”。
- **捎带确认**: 若双方都有数据要发，ACK 可以随数据一起带回；若无数据可捎带，也会单独发送 ACK。
- **初始序号**: TCP 两端通常随机选择初始序号，降低旧连接残留报文段被新连接误收的概率。

### 最可能怎么考
- **计算题**: A 发送 Seq=42, 长度=31 的数据报文给 B。B 收到后，回复的 ACK 号是多少？
  - **答案**：42 + 31 = **73**。（B 期望收到 A 的下一个字节编号为 73）。
- **填空题**: 一个 TCP 报文段的“首部长度”字段值为二进制的 `0111` (十进制 7)，则该 TCP 首部的实际长度是 **28** 字节，其中包含了 **8** 字节的选项部分。
- **判断题**: 如果接收方没有数据要发，就不会向发送方发送 ACK。（错误：ACK 可单独发送。）
- **计算题**: 两个连续 TCP 报文段序号分别为 90 和 110，则第一个报文段携带多少字节数据？若第一个丢失、第二个到达，接收方 ACK 号是多少？
  - **答案**：第一个报文段数据长度为 $110-90=20$ 字节。若第一个丢失，接收方仍期待字节 90，因此回复的 ACK 号是 **90**。

## 3.5.3 往返时间的估计与超时
### 重要性
★★★★
### 要背的边角料
- **SampleRTT (样本 RTT)**: 某报文段从发出到收到确认的时间；TCP 通常不为每个报文段都测一次。
- **平滑 RTT**: $EstimatedRTT = (1-\alpha)EstimatedRTT + \alpha SampleRTT$，EstimatedRTT 表示**估计 RTT**，常取 $\alpha=0.125$。
- **RTT 偏差**: $DevRTT = (1-\beta)DevRTT + \beta |SampleRTT-EstimatedRTT|$，DevRTT 表示**RTT 偏差**，常取 $\beta=0.25$。
- **EWMA (指数加权移动平均)**: EstimatedRTT 是指数加权移动平均，对最近样本权重更高。
- **公式**: $TimeoutInterval = EstimatedRTT + 4 \cdot DevRTT$，TimeoutInterval 表示**超时间隔**。
- **初始超时**: 推荐初始 TimeoutInterval 为 **1 秒**；发生超时时，TimeoutInterval 会加倍。
- **TimeoutInterval 判断陷阱**：不能只凭某一次 SampleRTT 直接判断当前 TimeoutInterval，它由平滑后的 EstimatedRTT 和 DevRTT 共同决定。
- **SampleRTT 陷阱**: 对重传报文段通常不计算 SampleRTT，避免无法判断 ACK 对应原始发送还是重传。
- **不测重传 RTT 的原因**：重传后的 ACK 可能确认原始发送，也可能确认重传副本，发送方无法判断真实样本 RTT；这就是 Karn/Partridge 思路的直觉。

### 最可能怎么考
- **计算题**: 计算新的 TimeoutInterval。
  - **答案解析**：先按 $EstimatedRTT=(1-\alpha)EstimatedRTT+\alpha SampleRTT$ 更新 RTT 估计，再按 $DevRTT=(1-\beta)DevRTT+\beta |SampleRTT-EstimatedRTT|$ 更新偏差，最后算 $TimeoutInterval=EstimatedRTT+4DevRTT$。例：旧 $EstimatedRTT=100ms$，旧 $DevRTT=10ms$，$SampleRTT=120ms$，$\alpha=0.125$，$\beta=0.25$，则新 $EstimatedRTT=102.5ms$，新 $DevRTT=11.875ms$，超时间隔约为 $150ms$。

## 3.5.4 可靠数据传输
### 重要性
★★★★
### 要背的边角料
- **可靠字节流目标**: 接收进程读到的数据应无损坏、无间隙、无冗余、按序。
- **单一定时器**: TCP 推荐只维护一个重传定时器，可理解为关联最早未确认报文段。
- **SendBase (发送基序号)**: 最早未被确认字节的序号；若 ACK 值 y 大于 SendBase，则 y 之前的字节均已被确认。
- **超时处理**: 发生超时后重传最早未确认报文段，并通常加倍 TimeoutInterval；收到新的确认后再按 RTT 估计公式更新。
- **TCP ACK 生成规则 1**: 按序报文段到达，且此前所有数据都已确认时，接收方可延迟 ACK，最多等待 **500 ms** 看是否有下一个按序报文段到达。
- **TCP ACK 生成规则 2**: 若已有一个按序报文段在等待 ACK，又到达另一个按序报文段，则立即发送一个累积 ACK。
- **TCP ACK 生成规则 3**: 若到达的报文段序号大于期望序号，说明出现间隔，接收方立即发送冗余 ACK，ACK 号为间隔低端。
- **TCP ACK 生成规则 4**: 若到达的报文段部分或完全填补接收间隔，并且从间隔低端开始，则立即发送 ACK。
- **隐式 NAK**: 3 个冗余 ACK 可被看成对后续缺失报文段的隐式否定确认。
- **快速重传**: 收到 **3 个冗余 ACK** 时立即重传。
- **为什么不是 1 个冗余 ACK 就重传**: 单个冗余 ACK 可能只是报文段乱序造成的，等待 3 个冗余 ACK 能降低误判和不必要重传的概率。
- **发送缓存与应用阻塞**: 如果应用写入 TCP 的速率超过 TCP 能发送或接收方能读取的速率，发送缓存会被填满，应用进程会被阻塞；这不是接收方 `rwnd` 单独能解释的全部现象。
- **TCP 与 GBN/SR**: TCP 累积确认、维护 `SendBase` 和 `NextSeqNum`，看起来像 GBN；但许多实现会缓存失序报文段，且快速重传通常最多重传一个报文段，因此又不像纯 GBN。
- **SACK (选择确认)**: 选择确认 (Selective Acknowledgment, SACK) 允许接收方选择性确认失序报文段；与选择重传结合时，TCP 更像 SR。教材结论：TCP 差错恢复最好看成 **GBN 与 SR 的混合体**。

### 最可能怎么考
- **填空题**: TCP 触发快速重传的条件是收到 **3** 个冗余 ACK。
- **简答题**: TCP 是 GBN 还是 SR？
  - **答案**：TCP 使用累积确认并维护最早未确认字节，像 GBN；但许多 TCP 会缓存失序报文段，快速重传常只重传缺失报文段，SACK 又支持选择确认，所以 TCP 更适合说成 GBN 和 SR 的混合体。

## 3.5.5 流量控制 (Flow Control)
### 重要性
★★★
### 要背的边角料
- **rwnd (接收窗口)**: 接收方通过窗口字段通知。
- **核心约束**: 发送方必须保证 `LastByteSent - LastByteAcked <= rwnd`。
- **窗口计算**: `rwnd = RcvBuffer - (LastByteRcvd - LastByteRead)`，即接收缓存剩余空间。
- **rwnd 会变化**：`rwnd` 取决于接收缓存剩余空间，应用读出数据、网络到达新数据都会改变它；不要把它当作整条连接中固定不变的值。
- **SendBase/LastByteRcvd 关系**：对 A 发往 B 的数据流，B 若按序收到到字节 x，会回 ACK x+1；A 收到这个 ACK 后，`SendBase` 才能推进到对应确认号。若存在失序缓存，ACK 号仍指向第一个缺失字节。
- **零窗口探测**: 发送 1 字节分组打破死锁。
- **UDP 对比**: UDP 不提供流量控制；若应用读取不及时，接收方套接字前的缓存可能溢出并丢弃报文段。

### 最可能怎么考
- **简答题**: TCP 流量控制是如何实现的？
  - **答案**：TCP 发送方会维护一个接收窗口 (`rwnd`) 变量。接收方通过将自己当前的剩余缓存空间大小放入 TCP 报文首部的 `rwnd` 字段发送给对方，发送方据此限制自己发送的数据量，确保发送方发出的**未确认数据量**永远不会超过 `rwnd`，从而避免撑爆接收方的缓存。

## 3.5.6 TCP 连接管理
### 重要性
★★★★★


### 要背的边角料
- **三次握手**: SYN -> SYN-ACK -> ACK。
- **握手字段**: 第 1 步 `SYN=1, seq=client_isn`；第 2 步 `SYN=1, ACK=1, ack=client_isn+1, seq=server_isn`；第 3 步 `ACK=1, ack=server_isn+1`。
- **握手目的**: 同步双方初始序号，确认双方收发能力，避免旧的失效连接请求造成错误连接。
- **四次挥手**: FIN -> ACK -> FIN -> ACK。
- **SYN/FIN 占序号**: SYN 和 FIN 虽不携带应用数据，但会消耗一个序号。
- **客户端典型状态**: `CLOSED -> SYN_SENT -> ESTABLISHED -> FIN_WAIT_1 -> FIN_WAIT_2 -> TIME_WAIT -> CLOSED`。
- **服务器典型状态**: `CLOSED -> LISTEN -> SYN_RCVD -> ESTABLISHED -> CLOSE_WAIT -> LAST_ACK -> CLOSED`。
-  408 状态机陷阱 (TIME_WAIT)**: 
    - 主动关闭方在发送最后一个 ACK 后，必须进入 **TIME_WAIT** 状态，等待 **2MSL (Maximum Segment Lifetime)**。
    - 为什么？1. 保证最后一个 ACK 如果丢失可以重传；2. 让网络中“迷失”的旧报文段彻底消亡，防止影响下一个具有相同四元组的新连接。
- **教材 TIME_WAIT 口径**: Kurose 第八版还写到 TIME_WAIT 的具体实现时间可为 **30 秒、1 分钟或 2 分钟**；408/经典口径常概括为 **2MSL**。
- **SYN 洪泛攻击**: 攻击者大量发送 SYN 但不完成第三次握手，耗尽服务器半开连接资源。
- **SYN cookie**: 服务器把源/目的 IP、端口和秘密数哈希进初始序号，不先分配连接状态；收到合法 ACK 后再建立连接。
- **端点鉴别边界**：UDP 服务器会把响应发往报文中的源 IP/端口，不能证明请求者真的在该地址；TCP 三次握手配合随机初始序号只能提供有限的“源地址可达”证明，不能抵抗中间人。
- **RST**: TCP 收到无法匹配套接字的报文段时可回 RST；UDP 目的端口无匹配时通常触发 ICMP (因特网控制报文协议) 差错报文。
- **nmap TCP 端口判断**: 收到 SYNACK (SYN+ACK 响应) 表示端口打开；收到 RST 表示主机可达但该 TCP 端口无应用监听；什么也收不到通常表示被防火墙阻挡。

### 最可能怎么考
- **简答题 1**: 为什么建立连接需要三次握手，而释放连接需要四次挥手？
  - **答案**：**握手三次**是为了同步双方的初始序号，并防止网络中延迟的、失效的连接请求突然到达服务端引发错误。**挥手四次**是因为 TCP 是全双工的，一方发送 FIN 仅仅代表“我没有数据要发送了”，但可能仍在接收数据；所以必须每个方向单独发送 FIN 和 ACK，共四次。
- **简答题 2**: 主动关闭 TCP 连接的一方为什么要等待 2MSL 时间后才真正释放连接？
  - **答案**：1. **确保最后一个 ACK 到达**：如果最后的 ACK 丢失，被动关闭方会重传 FIN，主动方在 2MSL 内可以再次发送 ACK；2. **清除网络旧报文**：确保本次连接产生的所有报文段都从网络中消失，不会与未来相同端口的新连接发生混淆。

# 3.6 拥塞控制原理
## 3.6.1 拥塞代价与控制方法
### 重要性
★★★★
### 要背的边角料
- **拥塞根因**: 太多源以过高速率发送数据，导致路由器缓存溢出和排队时延激增；重传能修复丢包症状，但不能消除拥塞原因。
- **三个速率符号**: `λin` 表示应用送入套接字的初始数据速率；`λ'in` 表示进入网络的供给载荷（初始数据 + 重传）；`λout` 表示接收端实际吞吐量。
- **情况 1：无限缓存单跳路由器**: 吞吐量最多接近每连接 `R/2`，但当发送速率接近链路容量时，排队时延趋于无穷大。
- **情况 2：有限缓存单跳路由器**: 缓存溢出会造成丢包，发送方必须重传；过早超时会让路由器转发不必要的重复分组，浪费带宽。
- **大缓存陷阱**：增加路由器缓存不一定提高有效吞吐量；固定超时下，排队时延变大可能诱发过早重传，动态超时能减少不必要重传，但过大的排队时延本身仍是代价。
- **情况 3：有限缓存多跳路径**: 分组若在下游路由器被丢弃，上游路由器已经消耗的传输容量全部浪费；重载极限下端到端吞吐量甚至可能趋近于 0。
- **拥塞代价**: 排队时延增加、丢包重传浪费资源、已在上游消耗的传输资源因下游丢包而浪费。
- **拥塞崩溃**: 供给载荷继续增加时，有效吞吐量反而下降，网络资源大量浪费在被丢弃或不必要重传的分组上。
- **端到端拥塞控制**: 端系统通过丢包和时延推断拥塞，例如 TCP。
- **网络辅助拥塞控制**: 路由器显式反馈拥塞信息。
- **网络辅助两种反馈路径**: 路由器直接向发送方反馈；或路由器标记经过的分组，再由接收方通知发送方。
- **ECN (显式拥塞通告)**: 网络层标记拥塞，TCP 使用 ECE 与 CWR 标志反馈和响应。
- **拥塞与流量控制区别**: 拥塞控制保护网络，避免把网络打爆；流量控制保护接收方，避免把接收缓存撑爆。

### 最可能怎么考
- **简答题**: 拥塞会造成哪些代价？
  - **答案**：拥塞会导致排队时延迅速增加；缓存溢出后发生丢包，发送方重传会浪费链路带宽；如果分组在多跳路径的下游被丢弃，上游已经消耗的传输容量也被浪费；当供给载荷继续增加时，有效吞吐量可能反而下降，形成拥塞崩溃。
- **简答题**: 比较端到端拥塞控制与网络辅助拥塞控制。
  - **答案**：
    1. **端到端拥塞控制**：网络层不提供显式支持。端系统（如 TCP）只能通过观察网络行为（如分组丢失、时延增加）来**推断**网络是否拥塞。
    2. **网络辅助拥塞控制**：网络层实体（路由器）直接或间接向端系统提供明确的拥塞反馈（例如通过一个特定的标志位指示发生了拥塞）。

# 3.7 TCP 拥塞控制
## 3.7.1 拥塞控制算法
### 重要性
★★★★★

 - **初始进入慢启动**：`cwnd=1 MSS`，`ssthresh=64 KB`，`dupACKcount=0`。 - **慢启动收到 new ACK**：`cwnd=cwnd+MSS`，表现为每个 RTT 近似翻倍；若 `cwnd >= ssthresh`，转入拥塞避免。 - **拥塞避免收到 new ACK**：`cwnd=cwnd+MSS*(MSS/cwnd)`，表现为每个 RTT 约增加 1 MSS。
 - **任一阶段 timeout**：`ssthresh=cwnd/2`，`cwnd=1 MSS`，`dupACKcount=0`，重传丢失报文段，回到慢启动。 
 - **dupACKcount3**：`ssthresh=cwnd/2`，`cwnd=ssthresh+3*MSS`，快速重传丢失报文段，进入快速恢复。
- **快速恢复收到 duplicate ACK**：`cwnd=cwnd+MSS`，允许时继续发送新报文段。
- **快速恢复收到 new ACK**：`cwnd=ssthresh`，`dupACKcount=0`，转入拥塞避免。

### 要背的边角料
- **拥塞窗口 (cwnd)**: 限制发送方速率的变量。`发送窗口 = min(cwnd, rwnd)`。
- **精确约束**: `LastByteSent - LastByteAcked <= min(cwnd, rwnd)`。
- **发送速率近似**: TCP 发送速率约为 `cwnd / RTT`。
- **丢包事件**: TCP 将超时或 **3 个冗余 ACK** 视为网络拥塞信号。
- **自计时 (self-clocking)**: ACK 到达说明网络正在交付数据，TCP 用 ACK 到达来触发窗口增长。
- **带宽探测**: ACK 到达就增加速率，丢包事件就降低速率，以分布式方式逼近可用带宽。
- **三条指导原则**: 丢失报文段意味着拥塞，应降低速率；确认报文段意味着网络正在交付数据，可增加速率；TCP 通过反复增速、遇丢包后回退来探测带宽。
- **为什么还要窗口控制**：超时后加倍 TimeoutInterval 只能降低重传频率，不能限制发送方持续注入新数据；拥塞窗口负责直接约束网络中的未确认数据量。
- **算法三部分**: 慢启动、拥塞避免是 TCP 强制部分；快速恢复是推荐部分。
- **阶段 1：慢启动 (Slow Start)**: `cwnd` 从 1 开始，每收到一个 ACK，cwnd 加 1（表现为每个 RTT 后 **指数倍增**），直到达到阈值 `ssthresh`。
- **慢启动结束条件**: 超时则 `ssthresh=cwnd/2` 且 `cwnd=1` 重新慢启动；`cwnd=ssthresh` 时转入拥塞避免；收到 3 个冗余 ACK 时快速重传并进入快速恢复。
- **阶段 2：拥塞避免 (Congestion Avoidance)**: 达到阈值后，每个 RTT 仅将 `cwnd` 增加 1 MSS（**线性增长**）。
- **拥塞避免按 ACK 增长**: 常见实现是每收到一个新 ACK，将 `cwnd` 增加约 `MSS*(MSS/cwnd)` 字节，所以一个 RTT 总共约增加 1 MSS。
- **快速恢复**: 快速重传后进入；经典教材口径中它是推荐部分，慢启动和拥塞避免是强制部分。
- **🚨 408 致命计算陷阱 (Reno vs Tahoe)**:
    - **遇到超时 (Timeout)**：不管是 Tahoe 还是 Reno，阈值 `ssthresh = cwnd/2`，然后 **cwnd 强制降为 1**，进入慢启动。
    - **遇到 3 个冗余 ACK (3 Duplicate ACKs)**：
        - **TCP Tahoe (旧版)**: 同样 `ssthresh = cwnd/2`，`cwnd = 1`。
        - **TCP Reno (考研常考)**: `ssthresh = cwnd/2`，然后 `cwnd = ssthresh + 3`，进入**快速恢复 (Fast Recovery)** 阶段。
- TCP Reno对拥塞控制的加性增、乘性减方法 （AIMD），
- **TCP CUBIC 核心思想**: 使用三次函数调整拥塞窗口；接近先前最大窗口 $W_{max}$ 时增长较慢，远离 $W_{max}$ 时增长较快。
- **TCP CUBIC 的直觉**: 快速回到上次拥塞前的发送速率附近，但在可能再次拥塞的 $W_{max}$ 附近谨慎探测。
- **TCP CUBIC 应用**: 适合高带宽时延积链路，是 Linux 常见默认拥塞控制算法。
- **TCP 吞吐宏观模型**: 经典 Reno 的发送速率大致在 $W/(2RTT)$ 到 $W/RTT$ 之间锯齿式变化，平均吞吐量常近似为 $0.75W/RTT$。
- **ssthresh 严谨口径**：教材计算题常用 `ssthresh=cwnd/2`；若问实际丢包时发送速率是否一定等于 `cwnd/RTT`，答案是否定的，因为应用受限或接收窗口受限时未必填满 `cwnd`，更严谨做法是按当前在途数据量的一半设置。
- **TCP 分岔 (TCP splitting)**：客户先连近处前端服务器，前端再用大窗口 TCP 连接后端数据中心；响应时间近似 `4RTT_FE + RTT_BE + 处理时间`，若前端离客户很近，可近似降为 `RTT_BE + 处理时间`。
- **空闲后重启风险**：TCP 长时间空闲后若直接沿用旧 `cwnd`，可能突然注入大量数据造成拥塞；若完全从 1 开始又可能太保守，因此实现通常会更谨慎地重启发送。

### 最可能怎么考
- **判断题**: TCP 超时时，`ssthresh` 被设置为原来 `ssthresh` 的一半。（错误：通常设置为发生拥塞时当前 `cwnd` 的一半。）
- **计算题/大题**: 设 TCP Reno 拥塞窗口 `cwnd` 初始为 16 MSS，在下一次发送时收到了 3 个冗余 ACK。请问此时的阈值 `ssthresh` 和新的 `cwnd` 分别变为多少？
  - **答案**：发生 3 个冗余 ACK。首先计算阈值：`ssthresh = cwnd / 2 = 16 / 2 = 8`。因为是 TCP Reno，执行快速恢复算法：新的 `cwnd = ssthresh + 3 = 8 + 3 = 11 MSS`。
- **简答题**: TCP CUBIC 为什么适合高速长距离链路？
  - **答案**：传统 TCP Reno 在高带宽、大时延链路上一旦窗口减半，线性增长太慢，难以迅速重新占满带宽。CUBIC 用三次函数增长窗口，远离 $W_{max}$ 时快速增长，接近 $W_{max}$ 时放慢探测，所以能兼顾带宽利用率和拥塞风险。

## 3.7.2 明确拥塞通告与基于时延的拥塞控制
### 重要性
★★
### 要背的边角料
- **ECN**: 明确拥塞通告，是网络辅助拥塞控制的一种形式。
- **ECN 标记位置**: 网络层在 IP 数据报服务类型字段中使用 ECN 比特标记拥塞。
- **ECE/CWR**: 接收方通过 TCP ACK 中的 ECE 比特回显拥塞；发送方减小窗口后用 CWR 比特说明已响应。
- **基于时延的拥塞控制**: 在真正丢包前，通过 RTT/排队时延增长提前感知拥塞。
- **TCP Vegas**: 以最小 RTT (`RTTmin`) 估计无拥塞吞吐量，并用实际吞吐量与无拥塞吞吐量的差距提前判断拥塞。
- **BBR (瓶颈带宽和往返传播时间)**: 以瓶颈带宽和 RTT 为核心估计路径状态，属于较新的拥塞控制思路。
- **BBR 直觉**: 目标是让瓶颈链路“刚好充满但不过满”，避免为了排长队而增加时延。

### 最可能怎么考
- **填空题**: 明确拥塞通告缩写为 **ECN**；TCP 中常配合 **ECE** 和 **CWR** 标志位使用。

## 3.7.3 公平性
### 重要性
★★
### 要背的边角料
- **AIMD (加性增、乘性减)**: TCP 趋向公平。
- **为什么必须乘性减**：AIMD 的公平性依赖“加性增大、乘性减小”；如果改成加性增大、加性减小 (AIAD)，通常不能像 AIMD 那样自然收敛到公平共享。
- **公平共享**: 两条条件相近的 TCP 连接共享同一瓶颈链路时，长期看通常趋向各占约一半瓶颈带宽。
- **UDP 不公平**: UDP 不受 TCP 拥塞控制约束，若持续高速发送，可能压制 TCP 连接。

### 最可能怎么考
- **简答题**: 为什么 AIMD 机制会使多个 TCP 连接趋向公平？
  - **答案**：多个连接共享瓶颈链路时，未拥塞阶段都按近似相同速率**加性增大**窗口；一旦发生拥塞，各连接又按比例**乘性减小**窗口。反复执行后，吞吐量大的连接会在拥塞时减少得更多，吞吐量小的连接逐步追上，因此总体趋向公平共享瓶颈带宽。
- **填空题**: 两条相似 TCP 连接共享速率为 R 的瓶颈链路，长期公平状态下每条连接大约获得 **R/2**。

# 3.8 运输层功能的演化
## 3.8.1 QUIC
### 重要性
★★★
### 要背的边角料
- **基础**: QUIC 是运行在 UDP 之上的应用层协议，在应用空间实现可靠传输、拥塞控制和 TLS (传输层安全) 安全。
- **流量占比边角**: 教材提到当时超过 **7%** 的因特网流量来自 QUIC。
- **设计目标**: QUIC 旨在提高安全 HTTP 的运输层服务性能，使用 UDP 作为底层运输层协议，并为精简版 HTTP/2 设计。
- **HTTP/3**: HTTP/3 将 QUIC 作为支撑协议。
- **连接 ID**: QUIC 连接状态包含源连接 ID 和目标连接 ID；连接 ID 写在 QUIC 首部中，IP 地址变化时可支持连接迁移。
- **加密**: QUIC 分组都是加密的；连接握手和鉴别/加密握手被结合起来。
- **握手优势**: QUIC 将连接握手与鉴别/加密握手结合，减少传统 TCP + TLS 的多个 RTT 建连开销。
- **多流复用**: 一个 QUIC 连接可复用多条流；每条流有流 ID，连接 ID 和流 ID 都在 QUIC 首部中。
- **流的定义**: QUIC 中的流是两个端点之间可靠、按序、双向数据交付的抽象；HTTP/3 中一个 Web 页面里的每个对象可对应不同流。
- **避免应用层 HOL (队首阻塞)**: QUIC 对每个流分别提供可靠、按序交付，一个流丢包不会阻塞其他流的交付。
- **拥塞控制**: QUIC 拥塞控制基于 TCP NewReno (新版 Reno 拥塞控制)，整体上要求 TCP-friendly。
- **更新优势**: QUIC 在应用层实现，可按“应用更新时间尺度”更新，比改 TCP/UDP 更快。

### 最可能怎么考
- **填空题**: QUIC 基于 **UDP** 实现，并支撑 HTTP/3。
- **简答题**: QUIC 为什么能缓解 HTTP 层队头阻塞？
  - **答案**：QUIC 在一个连接中复用多条流，并对每条流分别提供可靠、按序交付；某条流中的 UDP 报文丢失时，只阻塞该流的数据交付，不会阻塞其他流继续交付给应用。

### 第 3 章高频应用题：P37 GBN、SR、TCP 丢包对比
- **题目原型**：A 向 B 连续发送 5 个数据报文段，编号为 1、2、3、4、5，其中第 2 个报文段丢失。比较 GBN、SR 和 TCP（无延迟 ACK）在最终成功交付 5 个报文段前，A 总共发送多少个数据报文段，B 总共发送多少个 ACK，以及这些序号分别是什么。
- **统一约定**：
  - GBN/SR 中，`ACK n` 表示第 n 个分组已经被确认。
  - TCP 中，`ACK2` 表示“仍期待第 2 个数据段”，`ACK6` 表示“1 到 5 都已累计确认”。真实 TCP 用字节序号，考试若给出字节数，就把这里的段编号换成对应字节序号。

| 协议 | A 总共发送的数据报文段 | B 总共发送的 ACK | 关键原因 |
| :-- | :-- | :-- | :-- |
| GBN | 9 个：1、2(丢)、3、4、5、2、3、4、5 | 8 个：ACK1、ACK1、ACK1、ACK1、ACK2、ACK3、ACK4、ACK5 | 接收方不缓存失序分组，3、4、5 到达后都丢弃并重复确认 ACK1；发送方超时后从 2 开始回退重传 2、3、4、5。 |
| SR | 6 个：1、2(丢)、3、4、5、2 | 5 个：ACK1、ACK3、ACK4、ACK5、ACK2 | 接收方缓存失序分组 3、4、5，并分别确认；发送方只重传真正丢失的 2。 |
| TCP | 6 个：1、2(丢)、3、4、5、2 | 5 个：ACK2、ACK2、ACK2、ACK2、ACK6 | TCP 使用累积确认；收到 3、4、5 时连续返回冗余 ACK2，发送方收到 3 个冗余 ACK 后快速重传 2；2 到达后即可累计确认到 ACK6。 |

- **第二问：若所有协议的超时值都远大于 5RTT，谁最快？**
  - **答案：TCP 最快。**
  - **解析**：GBN 和 SR 都要等第 2 个报文段的定时器超时后才能重传；TCP 不必等超时，它能靠 3 个冗余 ACK 触发快速重传。SR 比 GBN 少重传，但仍卡在超时等待上；TCP 因为快速重传，所以成功交付 5 个报文段的时间最短。
- **背诵口诀**：GBN 丢一个，后面一串重传；SR 丢哪个，只补哪个；TCP 靠累积 ACK 和 3 个冗余 ACK 快速补洞。
