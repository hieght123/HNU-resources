# 第 4 章 网络层：数据平面

### 重要性
★★★★★
网络层是实现主机到主机通信的关键。数据平面专注于路由器的本地转发行为。

---

# 4.1 网络层概述
## 4.1.1 两个平面：数据平面与控制平面
### 重要性
★★★★★
### 要背的边角料
- **网络层分组名**: 网络层的分组称为 **数据报 (Datagram)**；运输层是报文段，链路层是帧。
- **路由器 vs 链路层交换机**: 二者都属于分组交换机；路由器按网络层 IP 数据报首部字段转发，是第 3 层设备；链路层交换机按链路层帧字段转发，是第 2 层设备。
- **转发 (Forwarding)**: 数据平面。分组从输入接口到输出接口的**本地**动作。
- **路由选择 (Routing)**: 控制平面。确定分组从源到目的地的**端到端**路径。
- **转发表 (Forwarding Table)**: 路由器根据到达分组首部字段查转发表，决定输出链路接口。
- **数据平面主功能**: 决定到达路由器输入链路的分组如何被转发到输出链路。
- **控制平面主功能**: 控制数据报沿源主机到目的主机端到端路径的转发方式，并涉及网络层组件的配置和管理。
- **传统控制平面**: 路由选择算法运行在每台路由器中，路由器之间交换路由选择报文。
- **SDN (软件定义网络)**: 逻辑集中式控制，将数据平面与控制平面分离。
- **SDN 控制平面**: 远程控制器计算并分发转发表，路由器主要执行数据平面转发。

### 最可能怎么考
- **填空/简答题**: 网络层分组叫什么？路由器和链路层交换机的根本区别是什么？
  - **答案**：网络层分组叫 **IP 数据报**。路由器根据网络层 IP 数据报首部字段转发，是三层设备；链路层交换机根据链路层帧首部字段转发，是二层设备。
- **填空题**: 数据平面决定分组从路由器**输入链路**到**输出链路**的本地转发；控制平面决定源到目的地的**端到端路径/转发方式**。
- **填空题**: 转发表的主要作用是根据分组首部字段决定应转发到哪个**输出链路接口**。
- **名词解释/简答题**: 请区分转发 (Forwarding) 与路由选择 (Routing)。
  - **答案**：
    1. **转发 (Forwarding)**：属于数据平面，是指将分组从路由器的输入链路接口转移到适当输出链路接口的**本地动作**。通常由硬件（交换结构）在纳秒级完成。
    2. **路由选择 (Routing)**：属于控制平面，是指确定分组从源到目的地所采用的**端到端路径**的网络范围过程。通常由软件在毫秒级计算。
- **简答题**: 传统路由器控制平面与 SDN 控制平面的区别是什么？
  - **答案**：传统方法中，每台路由器既执行数据平面转发，也运行控制平面的路由选择算法并与其他路由器交换路由信息；SDN 方法把控制平面从路由器中分离出来，由远程控制器计算并下发转发表，路由器主要根据表项执行转发。

## 4.1.2 网络服务模型
### 重要性
★★★
### 要背的边角料
- **因特网服务模型**: **尽力而为 (Best-effort)**。不保证交付、时延、吞吐量或顺序。
- **可能的服务模型**: 确保交付、时延上界、有序交付、最小带宽、安全服务等。
- **因特网的现实选择**: IP (网际协议) 层采用简单的尽力而为服务，把可靠性、拥塞控制等更多功能留给端系统和上层协议。

### 最可能怎么考
- **简答题**: 因特网网络层的服务模型是什么？它“不保证”哪些内容？
  - **答案**：因特网网络层采用**尽力而为服务**。它只尽力把数据报从源主机送到目的主机，但不保证一定交付、不保证按序交付、不保证端到端时延、不保证最小吞吐量，也不保证一定不会丢包。

# 4.2 路由器工作原理
## 4.2.1 输入端口处理与最长前缀匹配
### 重要性
★★★★★
### 要背的边角料
- **路由器四大组件**: 输入端口、交换结构、输出端口、路由选择处理器。
- **硬件/软件分工**: 输入端口、交换结构、输出端口属于数据平面，通常用硬件实现以达到线速；路由选择处理器偏控制平面，通常用软件实现，负责路由协议、管理和表项计算。
- **输入端口功能**: 物理层线路端接、链路层处理、查找/转发、必要时排队。
- **最长前缀匹配 (LPM)**: 匹配掩码位数最长的项。
- **查找机制**: 必须在纳秒级完成以匹配链路速率。
- **转发表副本**: 为了线速查找，输入端口通常保存转发表的本地副本，避免每个分组都去问中央路由选择处理器。
- **TCAM (三态内容可寻址存储器)**: 常用于高速转发表查找，把 32 比特 IP 地址放入 TCAM 后，可近似常数时间返回匹配的转发表项。
- **目的地转发**: 传统 IP 转发主要根据目的 IP 地址查找输出端口。
- **基于目的地转发的限制**: 如果只按目的 IP 地址转发，就不能把“同一目的地、不同源主机”的流量分到不同输出端口；这种更细的控制属于通用转发/流表的思路。

### 最可能怎么考
- **应用题/简答题**: 什么是“最长前缀匹配 (LPM)”原则？
  - **答案解析**：路由器将目的 IP 地址的二进制形式与转发表中的前缀进行匹配，选择**匹配位数最长**的那一项对应的接口进行转发。
  - **题目提醒**：考试中如果给出几个十进制 IP 掩码，一定要转换为二进制再数匹配位数，直接看十进制极易出错。
- **判断/填空题**: 传统基于目的地转发主要匹配**目的 IP 地址**；若题目要求按源主机区分同一目的地流量，单靠传统目的地转发表做不到。

## 4.2.2 交换结构
### 重要性
★★★★
### 要背的边角料
- **经内存交换**: 受限于内存带宽。
- **经总线交换**: 受限于总线带宽。
- **经互联网络 (Crossbar)**: 可**并行**转发去往不同输出端口的分组。
- **内存交换吞吐限制**: 分组通常要写入内存再读出内存，吞吐量受内存带宽限制。
- **总线交换吞吐限制**: 所有端口共享同一总线，一次只有一个分组能通过总线。
- **同时转发口径**: 经内存交换和经总线交换通常不能让两个分组同时穿过交换结构；纵横式交换可让去往不同输出端口的多个分组并行通过。
- **互联网络交换优势**: 若多个分组去往不同输出端口，可并行通过交换结构。
- **纵横式限制**: 纵横式交换结构在同一时隙可让不同输出端口并行接收分组，但一个给定输出端口同一时刻至多接收一个分组。

### 最可能怎么考
- **填空题**: **纵横式 (Crossbar)** 交换结构能够克服总线带宽限制，实现并行转发。
- **判断题**: 两个分组若去往**同一个输出端口**，即使采用纵横式交换结构，也不能在同一时隙同时送到该输出端口。
- **判断题**: 两个分组即使去往**不同输出端口**，若交换结构是共享总线或经内存交换，通常也不能在同一时刻同时通过交换结构。
- **简答题**: 比较经内存交换、经总线交换和经互联网络交换的性能瓶颈。
  - **答案**：1. **经内存交换**：受限于内存带宽，吞吐量最多为内存带宽的一半。2. **经总线交换**：所有端口共享一条总线，受限于总线带宽，一次只能传一个分组。3. **经互联网络 (Crossbar)**：非阻塞，只要多组分组的输出端口不同，即可并行转发。

## 4.2.3 输出端口处理
### 重要性
★★★
### 要背的边角料
- **输出端口功能**: 从交换结构接收分组，进行排队、调度、链路层封装和物理层发送。
- **输出排队原因**: 从交换结构到达输出端口的分组速率可能超过输出链路传输速率。
- **输出丢包原因**: 多个输入端口同时把分组送往同一个输出端口时，输出缓存可能被填满，之后到达的分组会被丢弃。
- **提高交换结构速率的局限**: 提高交换结构速率不能根本防止输出端口丢包，因为瓶颈是输出链路发送速率；交换结构更快只会让分组更快到达输出缓存。
- **缓存权衡**: 输出缓存太小会丢包，太大可能带来过长排队时延。
- **弃尾 (Drop-tail)**: 输出缓存满时，直接丢弃新到达分组，是最简单的丢包策略。

### 最可能怎么考
- **填空题**: 输出端口中决定下一个发送哪个分组的机制称为 **分组调度**。
- **简答题**: 为什么输出端口也会发生排队？
  - **答案**：如果多个输入端口的分组经过交换结构后同时到达同一个输出端口，而输出链路一次只能按链路速率发送分组，那么多余分组必须在输出端口缓存中等待，因此产生输出排队。

## 4.2.4 何时出现排队：HOL (队首阻塞)
### 重要性
★★★★★
### 要背的边角料
- **输入排队 (HOL 阻塞)**: 队列首部分组因目标口忙而等待，阻碍后续分组。
- **输出排队**: 交换速率 > 链路速率。
- **输入排队原因**: 交换结构速率不足，或队首分组目标输出端口被占用。
- **丢包位置**: 输入队列和输出队列都可能因缓存满而丢包。
- **消除输入端口丢包的口径**: 不靠无限缓存时，应让交换结构足够快，理想情况下至少达到所有输入端口线速总和，使分组不在输入端口长期积压；但若多个分组争用同一输出端口，拥塞会转移到输出端口。
- **AQM (主动队列管理)**: 在缓存完全填满前主动丢弃或标记分组，提前向发送方释放拥塞信号。
- **RED (随机早期检测)**: AQM 的经典例子，在队列过长前随机丢弃或标记部分分组。
- **缓存膨胀 (Bufferbloat)**: 缓存过大导致队列长期存在，吞吐量可能还可以，但排队时延会很高。
 - B(缓存数量)=RTT（平均往返时延）* C（链路容量）
### 最可能怎么考
- **名词解释**: 什么是 HOL 阻塞 (Head-of-the-Line blocking)？发生在路由器的哪个位置？
  - **答案**：即“线路前部阻塞”。在**输入端口**的队列中，排在队首的分组因为目标输出端口正在被其他端口占用而必须等待；这会导致跟在它后面的分组（即使它们的输出端口是空闲的）也被迫在队列中等待，从而降低了交换机的整体吞吐量。

## 4.2.5 分组调度
### 重要性
★★★★
### 要背的边角料
- **FIFO (先进先出)**: 先进先出。
- **优先权调度**: 高优先级先发；常见实现是**非抢占式**，即低优先级分组一旦开始传输，高优先级分组也要等它传完。
- **优先权调度风险**: 高优先级流量持续很多时，低优先级队列可能长期等待甚至饥饿。
- **优先权调度例子**: 网络运营者可能让实时语音/视频优先于电子邮件、文件传输，也可能让网络管理流量优先于普通应用流量。
- **循环/轮询调度 (Round Robin)**: 在不同类别队列之间轮流服务非空队列，避免只服务一个队列。
- **WFQ (加权公平排队)**: 按权重让不同类别获得不同链路份额，是加权版公平调度。
- **RR vs WFQ**: RR 默认各类轮流服务，近似平均分享；WFQ 按权重分享。若各类权重相同且分组大小影响忽略，WFQ 与 RR 表现近似相同。
- **网络中立性**: 分组调度能给不同类别流量不同服务等级，因此会引出政策问题；教材提到的三条规则是无阻塞、无节流/无限流、无付费优先。

### 最可能怎么考
- **简答题**: FIFO、优先权调度和 WFQ 的核心区别是什么？
  - **答案**：**FIFO** 按分组到达顺序发送，简单但不能区分业务优先级；**优先权调度**先发送高优先级队列，适合实时业务，但低优先级流可能被长期等待；**WFQ** 为不同类别分配权重，按权重分享链路带宽，在隔离业务的同时保持一定公平性。
- **填空题**: 为不同类别分配权重并按权重分享链路带宽的调度算法是 **WFQ**。
- **填空题**: 保证分组按到达顺序离开的调度规则是 **FIFO**。
- **填空题**: **RR** 默认轮流服务非空类别；当 WFQ 各类别权重相同且忽略分组大小差异时，WFQ 与 RR 的表现可近似相同。

# 4.3 网际协议：IPv4 (网际协议第 4 版), 编址, NAT (网络地址转换) 与 IPv6 (网际协议第 6 版)

## 4.3.1 IPv4 数据报格式
### 重要性
★★★★★
### 要背的边角料
- **字段顺序**: 版本、首部长度、服务类型、数据报长度、标识、标志、片偏移、TTL (生存时间)、协议、首部校验和、源 IP 地址、目的 IP 地址、选项、数据。
- **版本字段**: 占 **4 比特**；路由器通过版本号决定如何解释 IP 数据报。
- **首部长度字段**: 占 **4 比特**；用于确定数据字段从哪里开始。多数 IPv4 数据报没有选项，通常首部为 **20 字节**。
- **服务类型 (TOS)**: 用于区分不同类型 IP 数据报，例如实时流量和非实时流量。
- **数据报长度字段**: 表示整个 IP 数据报长度，即首部 + 数据；单位是 **字节**。字段长 **16 比特**，理论最大长度为 **65535 字节**。
- **标识、标志、片偏移**: 这三个字段与 IP 分片有关；一个大的 IP 数据报可被分成几个小的 IP 数据报独立转发，到目的地后重新组装。
- **TTL (生存时间)**: 每当一台路由器处理数据报时减 1；若减为 0，该数据报必须丢弃，避免长期路由环路。
- **协议字段**: 仅当 IP 数据报到达最终目的地时有用；值为 **6** 表示交给 TCP，值为 **17** 表示交给 UDP。
- **首部校验和**: 帮助路由器检测 IP 首部中的比特错误；因为 TTL 等字段会变化，所以每台路由器都要重新计算首部校验和。
- **校验和覆盖范围**: IPv4 首部校验和只覆盖 **IP 首部**；TCP/UDP 校验和覆盖运输层报文段，两者不对 IP 数据报中的共同字节重复计算。
- **分片触发与重组位置**: 当 IP 数据报太大、无法装入某条链路可承载的帧载荷时会被分片；IPv4 小数据报到达最终目的地后重新组装。
- **源和目的 IP 地址**: 源主机填入自己的 IP 地址和最终目的地 IP 地址；源主机通常通过 DNS 查找获得目的地址。
- **选项字段**: 允许扩展 IP 首部，但很少使用；选项会让首部长度可变，也会让路由器处理时间变化。
- **数据字段**: 通常承载 TCP 或 UDP 报文段，也可承载其他数据，如 ICMP 报文。无选项且承载 TCP 时，常见首部开销为 20B IP 首部 + 20B TCP 首部。
- **ICMP 层次口径**: ICMP 报文可放在 IP 数据报的数据字段中，但按书后 P 题口径，ICMP 通常归为网络层控制协议。

### 最可能怎么考
- **填空题**: IPv4 数据报长度字段单位是 **字节**，字段长 **16 比特**，理论最大长度为 **65535 字节**。
- **填空题**: 协议字段值 **6** 表示 TCP，**17** 表示 UDP。
- **填空题**: IPv4 首部校验和需要在每台路由器重新计算，原因之一是 **TTL 字段会变化**。
- **填空题**: IP 首部中的 **TTL** 字段可限制分组最多经过多少台路由器。
- **填空题**: IPv4 首部校验和只覆盖 **IP 首部**；TCP/UDP 校验和覆盖运输层报文段，二者没有共同覆盖的 IP 数据报字节。
- **填空/小计算**: 无选项 IP 首部 20B、TCP 首部 20B；若应用每次只发 40B 数据，则每个 IP 数据报的首部开销为 **40B**，应用数据占 **50%**。
- **填空/小计算**: 若 IP 数据报最大长度限制为 1500B 且 IP 首部为 20B，则该数据报最多承载 **1480B** 数据字段。
- **小计算题**: 要发送文件时，数据报数量 = `ceil(文件字节数 / 每个数据报可承载的数据字节数)`；在 1500B 限制、20B IP 首部时使用 `ceil(F / 1480)`。
- **简答题**: IPv4 的选项字段为什么会增加路由器处理复杂度？
  - **答案**：选项字段使首部长度可变，导致数据字段起始位置不固定；有些数据报需要处理选项，有些不需要，因此路由器处理不同数据报所需时间可能不同。

## 4.3.2 IPv4 编址
### 重要性
★★★★★
### 接口与 IP 地址
- **接口 (interface)**: 主机或路由器与物理链路之间的边界。
- **IP 地址分配对象**: 从技术上讲，一个 IP 地址与一个**接口**相关联，而不是与包括该接口的整台主机或路由器相关联。
- **路由器的 IP 地址**: 路由器有 IP 地址；准确说，路由器的每个接口通常各有一个 IP 地址。
- **IPv4 地址长度**: **32 比特**，等价为 4 字节。
- **点分十进制记法**: 每个字节用十进制写出并用点分隔；书上例子 `193.32.216.9` 的二进制为 `11000001 00100000 11011000 00001001`。
- **点分十进制转换例**: `223.1.3.27` 的 32 比特二进制为 `11011111 00000001 00000011 00011011`。

### 子网与 CIDR
- **子网 (subnet)**: 若一组接口不经过路由器即可互相到达，这些接口所在网络形成一个子网。因特网文献中，子网也称为 IP 网络或网络。
- **子网地址例子**: `223.1.1.0/24`、`223.1.2.0/24`、`223.1.3.0/24`。
- **子网掩码 / network mask**: 书上说 `/24` 这种 slash 记法有时称为子网掩码，表示 32 比特中最左侧 24 比特定义子网地址。
- **确定子网的方法**: 为了确定子网，可以分开主机和路由器的每个接口，得到若干隔离的网络岛；每个隔离网络就是一个子网。
- **CIDR (无类别域间路由选择)**: 形式为 `a.b.c.d/x`，其中 x 个最高比特构成网络部分，常称为前缀或网络前缀。
- **地址聚合**: 用单个网络前缀通告多个网络，也叫路由聚合或路由摘要；这样可减少外部路由器转发表长度。
- **最长前缀匹配**: 当路由器看到多个匹配前缀时，使用最长、最具体的地址前缀。
- **分类编址**: CIDR 之前，IP 地址网络部分被限制为 8、16 或 24 比特，分别对应 A、B、C 类网络；书上举例 C 类 `/24` 可容纳 254 台主机，B 类 `/16` 可支持 65534 台主机。
- **IP 广播地址**: `255.255.255.255`；目的地址为该值的数据报会交付给同一网络中的所有主机。

### 获取地址块与主机地址
- **获取地址块**: 组织通常从 ISP 获得一块地址，ISP 可从自己的较大地址块中分配连续地址块给组织。
- **ICANN**: IP 地址由 ICANN 管理；ICANN 还管理 DNS 根服务器，并把地址块分配给区域性因特网注册机构。
- **获取主机地址**: 主机地址可手工配置，但现在更多使用 DHCP。
- **DHCP (动态主机配置协议)**: 允许主机自动获得 IP 地址；也可让主机得知子网掩码、第一跳路由器地址（默认网关）和本地 DNS 服务器地址。
- **DHCP 性质**: 客户-服务器协议；因其自动配置能力，也称即插即用协议或零配置协议。
- **DHCP 中继代理**: 若子网内没有 DHCP 服务器，需要 DHCP 中继代理，通常是一台路由器。
- **DHCP 四步**: DHCP 发现、DHCP 提供、DHCP 请求、DHCP ACK。
- **DHCP 发现**: 客户向 UDP 端口 **67** 发送 DHCP 发现报文；IP 源地址用 `0.0.0.0`，目的地址用广播地址 `255.255.255.255`。
- **DHCP 提供**: 服务器用 DHCP 提供报文响应，报文包含事务 ID、推荐 IP 地址、网络掩码和 IP 地址租用期。
- **DHCP ACK**: 服务器确认请求后，客户即可在租用期内使用分配到的 IP 地址，并可更新租用。

### 最可能怎么考
- **填空题**: IP 地址实际与 **接口** 相关联。
- **填空题**: 路由器的 IP 地址数量取决于它的**接口数量**；通常每个接口一个 IP 地址。
- **填空题**: `223.1.3.27` 写成二进制是 `11011111 00000001 00000011 00011011`。
- **填空题**: `/24` 记法有时称为 **子网掩码**，表示最左侧 **24** 比特定义子网地址。
- **填空题**: 管理 IP 地址空间并管理 DNS 根服务器的机构是 **ICANN**。
- **填空题**: DHCP 又称为 **即插即用协议** 或 **零配置协议**。
- **填空题**: DHCP 可提供 IP 地址、子网掩码、默认网关和 **本地 DNS 服务器地址**。
- **填空/小计算**: 源主机和目的主机之间若有 3 台路由器，且不考虑分片，一条路径上会经过 **8 个接口**，中间路由器会查 **3 次转发表**。
- **简答题**: DHCP 协议的四个交互步骤分别是什么？
  - **答案**：DHCP 发现 -> DHCP 提供 -> DHCP 请求 -> DHCP ACK。

## 4.3.3 网络地址转换 (NAT)
### 重要性
★★★★
### 要背的边角料
- **NAT 动机**: SOHO 或家庭网络中有多台设备，若每台设备都需要公网地址，会加重 IPv4 地址分配压力；NAT 允许家庭网络对外表现为具有单一 IP 地址的单一设备。
- **专用网络地址例子**: 书上举 `10.0.0.0/8` 是保留的专用地址空间之一；家庭网络例子中 4 个接口具有相同网络地址 `10.0.0.0/24`。
- **专用地址边界**: `10.0.0.0/24` 这样的地址只在给定网络中有意义，不能作为全球因特网上转发分组的源地址或目的地址。
- **NAT 对外表现**: 所有离开家庭路由器的报文源 IP 都是 NAT 路由器 WAN 侧 IP，如 `138.76.29.7`；所有进入家庭网络的报文目的 IP 也是这个地址。
- **NAT 与 DHCP**: 通常路由器从 ISP 的 DHCP 服务器获得自己的公网地址，同时路由器运行 DHCP 服务器，为家庭网络主机提供地址。
- **家庭无线路由器口径**: ISP 通常只给无线路由器的 WAN 侧分配一个公网 IP；路由器给家中主机分配内网地址（通常通过 DHCP），并用 NAT 让这些主机共享该公网 IP。
- **NAT 转换表**: NAT 路由器用转换表记录 WAN 侧端口与 LAN 侧 IP/端口的映射。
- **书上例子**: 内网主机 `10.0.0.1` 用源端口 `3345` 请求 Web 服务器 `128.119.40.186:80`；NAT 将源 IP 改为 `138.76.29.7`，将源端口改为 `5001`，并记录转换表项。
- **端口数量口径**: 因为端口号字段为 16 比特，NAT 可支持超过 60000 个并行使用单个 WAN 侧 IP 地址的连接。
- **NAT 表项粒度**: NAT 转换表按外侧端口映射到内侧 IP/端口；并行 TCP 连接通常需要各自的映射表项。
- **NAT 后主机数检测的 P 题假设**: 若各主机生成的 IP 标识号顺序递增，且可观察 NAT 向外发出的分组，可从多个交错的独立标识号序列粗略推断 NAT 后主机数量；若标识号随机化，该方法失效。
- **NAT 问题**: 端口号本来用于进程寻址，NAT 把端口用于区分内部主机，会给家庭网络中的服务器和 P2P 对等方接受入连接带来问题。
- **NAT 穿越与 UPnP**: NAT traversal 工具和 UPnP 可缓解入连接问题。
- **体系结构争议**: NAT 修改 IP 地址甚至端口号，违背主机直接彼此通信的端到端思想，也使 NAT 成为不同于传统路由器的中间盒。

### 最可能怎么考
- **简答题**: 家庭中多台 PC 如何通过一个无线路由器接入只有一个公网 IP 的 ISP 连接？
  - **答案**：无线路由器从 ISP 获得一个 WAN 侧公网 IP，同时在家庭网络内部通常通过 DHCP 给各主机分配内网地址；这些主机访问外网时由路由器执行 NAT，共享同一个公网 IP。
- **填空题**: 专用网络地址只在给定网络内部有意义，具有专用地址的源/目的数据报不会被当作普通公共因特网数据报转发。
- **填空题**: NAT 转换表的核心映射是 **WAN 侧端口 -> LAN 侧 IP 地址/端口**；3 台主机各有 2 条并行 TCP 连接时，可形成 **6** 条映射表项。
- **简答题**: 若题目假设 IP 标识号顺序递增，如何粗略判断 NAT 后主机数量？
  - **答案**：观察 NAT 向外发出的 IP 分组标识号序列；不同内部主机可能形成不同的递增序列，交错序列的条数可作为 NAT 后主机数的线索。若标识号随机化，这个方法基本失效。
- **简答题**: NAT 为什么能让家庭网络共享一个公网 IP？
  - **答案**：NAT 路由器对外使用自己的 WAN 侧公网 IP；内网主机发往外网时，NAT 改写源 IP 和源端口，并在转换表中记录映射。回包到达公网 IP 和对应端口时，NAT 查表改写目的 IP 和端口，再转发给内部主机。
- **应用题**: 书上 NAT 例子中，`10.0.0.1:3345` 发往 `128.119.40.186:80` 时 NAT 如何改写？
  - **答案**：NAT 把源地址/端口改为 `138.76.29.7:5001`，并记录 `5001 -> 10.0.0.1:3345`；服务器回包到 `138.76.29.7:5001` 后，NAT 查表改回 `10.0.0.1:3345`。

## 4.3.4 IPv6
### 重要性
★★★★
### IPv6 数据报格式
- **首要动机**: IPv4 的 32 比特地址空间即将用尽，IPv6 将地址长度从 **32 比特**增加到 **128 比特**。
- **任播地址**: IPv6 除单播和多播外，引入任播地址，可把数据报交付给一组主机中的任意一个。
- **简化高效的首部**: IPv6 形成 **40 字节定长首部**，使路由器能更快处理 IP 数据报。
- **流标签**: 用于给属于特殊流的分组加标签，这些流可能要求特殊处理，如非默认服务质量或实时服务。
- **字段顺序**: 版本、流量类型、流标签、有效载荷长度、下一个首部、跳限制、源地址、目的地址、数据。
- **版本字段**: 占 **4 比特**；IPv6 的字段值为 **6**。
- **有效载荷长度**: 占 **16 比特**，给出 IPv6 定长 40 字节首部后面的字节数量。
- **下一个首部**: 标识数据字段要交付给哪个协议，如 TCP 或 UDP；使用与 IPv4 协议字段相同的值。
- **跳限制**: 每台转发路由器将该字段减 1；若减到 0，数据报被丢弃。
- **源地址和目的地址**: IPv6 地址为 **128 比特**。

### IPv6 中不再有的 IPv4 字段
- **分片/重新组装**: IPv6 不允许中间路由器进行分片和重新组装；若数据报太大，路由器丢弃该数据报并向发送方返回“分组太大”的 ICMP 差错报文。
- **首部检验和**: IPv6 去掉首部检验和，以减少路由器处理负担；IPv4 中 TTL 改变会导致每跳重新计算首部检验和。
- **选项**: 选项字段不再是标准 IP 首部的一部分；它可能出现在由“下一个首部”指出的位置上，因此基本首部保持 40 字节定长。

### 从 IPv4 到 IPv6 的迁移
- **迁移困难**: IPv6 使能系统可向后兼容 IPv4，但已部署的 IPv4 系统不能处理 IPv6 数据报；让全网在某个标志日同时升级不可行。
- **建隧道 (tunneling)**: 两个 IPv6 节点若中间经过 IPv4 路由器，可把整个 IPv6 数据报放入 IPv4 数据报的数据字段中穿越 IPv4 区域。
- **隧道中间路由器**: 中间 IPv4 路由器只按普通 IPv4 数据报转发，不知道其中封装了完整 IPv6 数据报。
- **IPv6 隧道协议号**: 隧道接收端通过 IPv4 首部协议号字段为 **41** 判断其有效载荷是 IPv6 数据报。

### 最可能怎么考
- **填空题**: IPv6 地址长度为 **128 比特**，基本首部为 **40 字节**定长。
- **填空题**: IPv6 的下一个首部字段作用类似 IPv4 的 **协议字段**；跳限制字段作用类似 IPv4 的 **TTL**。
- **填空题**: IPv6 中仍有**版本、源地址、目的地址**等基本字段；IPv4 的协议字段对应 IPv6 的**下一个首部**字段，IPv4 的 TTL 对应 IPv6 的**跳限制**字段。
- **填空题**: IPv6 去掉了 IPv4 首部中的**首部检验和**，并且不允许中间路由器进行**分片/重组**。
- **简答题**: IPv6 为什么去掉首部检验和和路由器分片？
  - **答案**：首部检验和会增加每跳处理开销，IPv4 还因 TTL 改变需要每跳重算；路由器分片和重组也耗时。IPv6 把这些负担移出中间路由器，使转发更快。
- **简答题**: IPv4 到 IPv6 的隧道机制是什么？
  - **答案**：隧道发送端把整个 IPv6 数据报封装进 IPv4 数据报的数据字段，中间 IPv4 路由器按普通 IPv4 转发；隧道接收端看到 IPv4 协议号 41 后取出内部 IPv6 数据报继续转发。
- **判断/简答题**: 可以把 IPv4 隧道理解成连接两个 IPv6 节点的逻辑链路；IPv6 数据报作为 IPv4 数据报的有效载荷穿过该逻辑链路。

# 4.4 泛化转发和 SDN
### 重要性
★★★★
### 教材主线
- **基于目的地转发的本质**: 在 4.2.1 中，传统转发可看成两步：查找目的 IP 地址，即“匹配”；把分组送到特定输出端口，即“操作”。
- **泛化转发 (generalized forwarding)**: 把“匹配加操作”推广到多个首部字段。匹配字段可来自链路层、网络层和运输层；操作可包括转发、负载均衡、重写首部、丢弃、送往特定服务器继续处理等。
- **分组交换机**: 因为泛化转发可同时使用链路层和网络层字段，书上后面更倾向称这些设备为**分组交换机**，而不只说第三层路由器或第二层交换机。
- **远程控制器**: 每台分组交换机中有一张匹配加操作表，该表由远程控制器计算、安装和更新。
- **OpenFlow 地位**: 教材用 OpenFlow 1.0 说明匹配加操作抽象。OpenFlow 中的匹配加操作转发表称为**流表 (flow table)**。
- **流表项三部分**: 首部字段值集合、计数器集合、操作集合。流表本质上是一个 API，可通过配置流表来编程每台分组交换机的行为。

## 4.4.1 匹配
### 要背的边角料
- **匹配字段数量口径**: OpenFlow 1.0 可匹配 **11 个分组首部字段 + 入端口 ID**，也就是图 4-29 中的 12 个值。
- **可匹配字段顺序**: 入端口、源 MAC、目的 MAC、以太网类型、VLAN ID、VLAN 优先权、IP 源地址、IP 目的地址、IP 协议、IP TOS、TCP/UDP 源端口、TCP/UDP 目的端口。
- **跨层匹配**: OpenFlow 匹配字段来自链路层、网络层和运输层，书上说这相当明显地违反了严格分层原则。
- **通配符**: 流表项可使用通配符，例如 `128.119.*.*` 可匹配前 16 比特为 `128.119` 的地址字段。
- **优先级匹配**: 一个分组若匹配多个流表项，执行最高优先级表项对应的操作。
- **未匹配处理**: 未匹配任何流表项的分组可能被丢弃，也可能被发送给远程控制器进一步处理。
- **OpenFlow 硬件匹配**: 与传统转发表类似，首部字段匹配常用 TCAM 快速实现。
- **不可匹配陷阱**: OpenFlow 1.0 不能按所有 IPv4 字段匹配，例如不能按 TTL、数据报长度、标识、标志、片偏移、首部校验和等字段匹配。

### 最可能怎么考
- **填空题**: OpenFlow 1.0 可匹配 **11 个分组首部字段 + 入端口 ID**。
- **填空题**: OpenFlow 1.0 可匹配的 IP 字段例子有 **源 IP 地址、目的 IP 地址、协议字段、TOS 字段**；不能匹配的 IPv4 字段例子有 **TTL、数据报长度、标识、标志、片偏移、首部校验和**。
- **判断题**: 一个分组匹配多个流表项时，采用**最高优先级**表项对应的操作。

## 4.4.2 操作
### 要背的边角料
- **操作列表**: 每个流表项都有零个或多个操作；若有多个操作，就按表中规定的次序执行。
- **转发**: 可转发到某个物理输出端口，也可广播到所有端口（到达端口除外），或通过端口集合多播；还可封装后发送给远程控制器。
- **丢弃**: 没有操作的流表项表示匹配分组应当被丢弃。
- **修改字段**: 在分组转发到输出端口之前，可重写图 4-29 中除 IP 协议字段外的二、三、四层首部字段。

### 最可能怎么考
- **填空/简答题**: OpenFlow 的典型操作包括 **转发、丢弃、修改字段、复制/广播/多播、发送给控制器**。
- **填空题**: 没有操作的流表项表示匹配分组被**丢弃**。

## 4.4.3 运行中的匹配加操作的 OpenFlow 例子
### 要背的边角料
- **例 1：简单转发**: 让来自 `h5/h6` 且发往 `h3/h4` 的流量从 `s3 -> s1 -> s2` 转发，避开 `s3` 和 `s2` 的直接链路。
- **例 2：负载均衡**: 来自 `h3` 发往 `10.1.*.*` 的流量走 `s2 -> s1`，来自 `h4` 发往 `10.1.*.*` 的流量走 `s2 -> s3 -> s1`。书上明确说这种行为不能通过基于 IP 的目的地转发取得。
- **例 3：防火墙**: `s2` 只转发来自 `10.3.*.*` 且发往 `h3/h4` 的流量；如果没有其他表项，则其他来源的流量不会被转发到 `s2` 相连主机。
- **有限可编程性**: 匹配加操作流表是一种有限形式的可编程性；更丰富的可编程分组处理语言例子是 **P4**。
- **转发表 vs OpenFlow 流表**: 基于目的地转发的转发表主要保存目的地址前缀到输出端口的映射；OpenFlow 流表项包含匹配字段、计数器和操作，能表达更一般的转发、丢弃、修改和复制等行为。

### 最可能怎么考
- **简答题**: 泛化转发与传统基于目的地址的转发有什么不同？
  - **答案**：传统 IP 转发主要根据目的 IP 地址查找输出端口；泛化转发可以匹配入端口、MAC 地址、IP 地址、协议号、运输层端口等多个字段，并执行转发、丢弃、修改、复制、发送控制器等多种操作。
- **简答题**: 基于目的地转发的转发表与 OpenFlow 流表有什么区别？
  - **答案**：传统转发表主要把目的 IP 地址前缀映射到输出端口；OpenFlow 流表由匹配字段、计数器和操作组成，可匹配多层首部字段，并执行转发、丢弃、修改、复制、送控制器等多种操作。
- **填空/简答题**: 用 OpenFlow 实现防火墙，本质是按源/目的地址、协议或端口等字段**匹配**，再执行**转发或丢弃**操作。
- **填空题**: 匹配加操作流表是一种有限形式的**可编程性**；教材提到更丰富的可编程分组处理语言是 **P4**。

# 4.5 中间盒
### 重要性
★★★☆（第八版新增边角，适合出简答/填空）
### 要背的边角料
- **中间盒 (Middlebox)**: 位于源主机和目的主机之间的数据路径上，执行普通 IP 路由器标准转发功能之外的其他功能的设备。
- **三类服务**:
  1. **NAT 转换**：重写 IP 地址和运输层端口号。
  2. **安全服务**：防火墙、入侵检测系统 (IDS)、入侵防止系统 (IPS)、深度分组检测 (DPI)。
  3. **性能增强**：内容缓存、压缩、负载均衡。
- **架构争议**: 中间盒常检查或修改网络层、运输层甚至应用层信息，打破传统“网络核心只转发、端系统处理复杂功能”的端到端原则。
- **NFV (网络功能虚拟化)**: 用通用硬件和软件实现专用中间盒功能，降低专用硬件、独立管理和升级成本。

### 最可能怎么考
- **简答题**: 什么是中间盒？举出三类典型功能。
  - **答案**：中间盒是位于源主机和目的主机之间的数据路径上，执行普通 IP 路由器转发以外功能的设备。典型功能包括 NAT 地址/端口转换，防火墙、IDS/IPS、DPI 等安全服务，以及缓存、压缩、负载均衡等性能增强服务。
- **简答题**: 为什么说中间盒会挑战端到端原则？
  - **答案**：端到端原则倾向于把复杂功能放在端系统中，网络核心主要负责简单转发。但中间盒会在网络路径中检查、修改或过滤分组，例如 NAT 修改 IP 地址和端口，防火墙按首部字段阻挡流量，DPI 检查有效载荷。这使网络内部不再只是透明转发者。
- **简答题**: NFV 想解决中间盒部署中的什么问题？
  - **答案**：传统中间盒常依赖专用硬件、专用软件和单独管理技能，部署、升级和维护成本高。NFV 希望用通用计算/网络/存储硬件加软件来实现这些网络功能，使中间盒功能更容易部署、迁移和升级。

### 截图 1：按主机数确定子网大小

**题目**：一个自制系统内主机数为 50 台，该系统分配到的 IP 地址块为 `192.168.18.0/24`。给出每一个局域网的子网网络地址、子网掩码以及最小和最大可用 IP 地址。

**做法**：

- 50 台主机需要满足 `2^h - 2 >= 50`。
- `h = 6`，因为 `2^6 - 2 = 62`。
- 所以前缀为 `/26`，子网掩码为 `255.255.255.192`。
- `/26` 的块大小是 `64`，所以网络地址每次加 64。

| 子网 | 网络地址 | 子网掩码 | 可用 IP 范围 | 广播地址 |
|---|---|---|---|---|
| 1 | `192.168.18.0/26` | `255.255.255.192` | `192.168.18.1 - 192.168.18.62` | `192.168.18.63` |
| 2 | `192.168.18.64/26` | `255.255.255.192` | `192.168.18.65 - 192.168.18.126` | `192.168.18.127` |
| 3 | `192.168.18.128/26` | `255.255.255.192` | `192.168.18.129 - 192.168.18.190` | `192.168.18.191` |
| 4 | `192.168.18.192/26` | `255.255.255.192` | `192.168.18.193 - 192.168.18.254` | `192.168.18.255` |

### 截图 2：按子网数量等分

**题目**：IP 地址块为 `145.12.23.64/26`，划分为 5 个子网，写出划分后各个子网的最大和最小 IP 地址。

**做法**：

- `/26` 一共有 `64` 个地址。
- 要划出 5 个子网，借位数满足 `2^n >= 5`，所以 `n = 3`。
- 新前缀是 `/26 + 3 = /29`。
- `/29` 每块 8 个地址，6 个可用地址，子网掩码为 `255.255.255.248`。

| 子网 | 网络地址 | 最小可用 IP | 最大可用 IP | 广播地址 |
|---|---|---|---|---|
| 1 | `145.12.23.64/29` | `145.12.23.65` | `145.12.23.70` | `145.12.23.71` |
| 2 | `145.12.23.72/29` | `145.12.23.73` | `145.12.23.78` | `145.12.23.79` |
| 3 | `145.12.23.80/29` | `145.12.23.81` | `145.12.23.86` | `145.12.23.87` |
| 4 | `145.12.23.88/29` | `145.12.23.89` | `145.12.23.94` | `145.12.23.95` |
| 5 | `145.12.23.96/29` | `145.12.23.97` | `145.12.23.102` | `145.12.23.103` |

> [!note]
> 这个 `/26` 实际可划出 8 个 `/29` 子网。题目只要求 5 个，写前 5 个即可。

### 截图 3：VLSM 按部门主机数分配

**题目**：某公司分配到的 IP 地址为 `172.20.0.0/22`。公司下设 4 个部门：总部 500 台，销售部 200 台，人力资源部 50 台，法务部 20 台。按照最合适主机数的子网大小原则划分子网，给出每个部门局域网的子网网络地址、子网掩码以及最小和最大可用 IP 地址。

**做法**：

- 总地址块 `172.20.0.0/22` 范围是 `172.20.0.0 - 172.20.3.255`。
- VLSM 必须先分大的，再分小的，避免小块切碎大块空间。

| 部门 | 主机数 | 最小满足条件 | 前缀 | 子网掩码 | 网络地址 | 可用 IP 范围 | 广播地址 |
|---|---:|---|---|---|---|---|---|
| 总部 | 500 | `2^9 - 2 = 510` | `/23` | `255.255.254.0` | `172.20.0.0/23` | `172.20.0.1 - 172.20.1.254` | `172.20.1.255` |
| 销售部 | 200 | `2^8 - 2 = 254` | `/24` | `255.255.255.0` | `172.20.2.0/24` | `172.20.2.1 - 172.20.2.254` | `172.20.2.255` |
| 人力资源部 | 50 | `2^6 - 2 = 62` | `/26` | `255.255.255.192` | `172.20.3.0/26` | `172.20.3.1 - 172.20.3.62` | `172.20.3.63` |
| 法务部 | 20 | `2^5 - 2 = 30` | `/27` | `255.255.255.224` | `172.20.3.64/27` | `172.20.3.65 - 172.20.3.94` | `172.20.3.95` |

剩余地址从 `172.20.3.96` 到 `172.20.3.255`，可留作扩展。

**口诀**：人数从大到小排，找 `2^h - 2`，换成前缀，按块大小连续分配。


### 书后 P18：NAT 转换表

**题目**：图 4-25 中，ISP 给路由器分配地址 `24.34.112.235`，家庭网络地址是 `192.168.1.0/24`。为家庭网络接口分配地址，并假定每台主机有两个到 `128.119.40.86:80` 的 TCP 连接，写 NAT 转换表 6 项。

一种可行地址分配：

- 主机 1：`192.168.1.1`
- 主机 2：`192.168.1.2`
- 主机 3：`192.168.1.3`
- 路由器 LAN 口：`192.168.1.4`
- 路由器 WAN 口：`24.34.112.235`

一种可行 NAT 表：

| WAN 侧 | LAN 侧 |
|---|---|
| `24.34.112.235:4000` | `192.168.1.1:3345` |
| `24.34.112.235:4001` | `192.168.1.1:3346` |
| `24.34.112.235:4002` | `192.168.1.2:3445` |
| `24.34.112.235:4003` | `192.168.1.2:3446` |
| `24.34.112.235:4004` | `192.168.1.3:3545` |
| `24.34.112.235:4005` | `192.168.1.3:3546` |

**改写规则**：

- 出站：源地址/端口从 `192.168.1.x:端口` 改成 `24.34.112.235:公网端口`。
- 入站：目的地址/端口从 `24.34.112.235:公网端口` 查表改回 `192.168.1.x:端口`。

### 书后 P21：OpenFlow 转发规则

**题目**：在图 4-30 的 `s2` 中写流表，实现指定转发行为。

考试够用写法：

| 匹配条件 | 动作 |
|---|---|
| `h5/h6 -> h1/h2`，从端口 1 到达 | `Forward(2)` |
| `h1/h2 -> h5/h6`，从端口 2 到达 | `Forward(1)` |
| 目的地址为 `h3` | `Forward(3)` |
| 目的地址为 `h4` | `Forward(4)` |
| `h3 -> h4` | `Forward(4)` |
| `h4 -> h3` | `Forward(3)` |

写答案时把主机换成 IP 前缀即可：

- `h1/h2` 属于 `10.1.*.*`
- `h3/h4` 属于 `10.2.*.*`
- `h5/h6` 属于 `10.3.*.*`

### 书后 P24：OpenFlow 防火墙规则

**题目**：在图 4-30 的 `s2` 中定义防火墙流表，使目的地为 `h3/h4` 的数据报只按题目规则通过。没有匹配项就默认丢弃。

考试够用写法：

| 防火墙要求                     | 允许规则                                                                      |
| ------------------------- | ------------------------------------------------------------------------- |
| 只允许 `h1` 和 `h6` 到 `h3/h4` | `src=h1 or h6, dst=h3 -> Forward(3)`；`src=h1 or h6, dst=h4 -> Forward(4)` |
| 只允许 TCP 到 `h3/h4`         | `protocol=TCP, dst=h3 -> Forward(3)`；`protocol=TCP, dst=h4 -> Forward(4)` |
| 只允许到 `h3`                 | `dst=h3 -> Forward(3)`                                                    |
| 只允许 `h1` 到 `h3` 的 UDP     | `src=h1, dst=h3, protocol=UDP -> Forward(3)`                              |

关键口径：防火墙不是“写拒绝项”为主，而是**只写允许项，其他未匹配流量默认丢弃**。

### 王道综合应用 01：IPv4 分片

**题目原型**：4000B IP 数据报，IP 首部 20B，MTU 1500B，计算分片数、每片长度、片偏移和 MF。

**做法**：

- 原数据部分：`4000 - 20 = 3980B`
- 每片最多承载数据：`1500 - 20 = 1480B`
- 片偏移单位是 8B，`1480 / 8 = 185`

| 分片 | 数据长度 | 总长度 | 片偏移 | MF |
|---|---:|---:|---:|---:|
| 1 | 1480 | 1500 | 0 | 1 |
| 2 | 1480 | 1500 | 185 | 1 |
| 3 | 1020 | 1040 | 370 | 0 |

第三片偏移：`(1480 + 1480) / 8 = 370`。最后一片 `MF = 0`，前两片 `MF = 1`。
